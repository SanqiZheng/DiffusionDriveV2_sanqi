# 13 实现观察、潜在问题与二次开发建议

这一篇不是“挑刺”。

它的目的有两个：

1. 帮你区分论文方法和当前工程实现的边界
2. 帮你找出最适合二次开发、复用和重构的切入点

## 1. 总体观察

当前仓库已经把论文主方法跑通了，但它明显保留了大量研究代码特征：

- 一部分地方写得非常直接
- 有不少硬编码
- 有一些遗留接口没有完全清理
- 某些数学对象在论文与代码之间不是严格一一对应

所以这份工程非常适合“学习算法”，但如果你要迁移到自己项目里，建议先做结构清理。

---

## 2. 与论文不一致或不完全一致的关键点

## 2.1 论文里的固定 `L = L_RL + \lambda L_IL`，代码里是动态权重

论文表达：

- `L = L_RL + λ L_IL`

代码实际：

- `IL_loss_b` 是多步轨迹 L1
- `il_weight` 会根据 batch 里是否存在正优势样本，在 `1.0` 和 `0.1` 之间切换

影响：

- 训练行为比论文公式更“自适应”
- 但也更难从论文直接推断真实训练曲线

## 2.2 论文把 policy 写成高斯分布，代码是乘性噪声近似实现

在 `DDIMScheduler_with_logprob.step` 里：

- 样本更新使用了乘性噪声
- `prev_sample = prev_sample_mean * variance_noise_mul + ...`

但 `log_prob` 的写法更像标准高斯残差：

- `-((prev_sample - prev_sample_mean)^2) / (2 * std^2)`

问题：

- 这并不是乘性噪声的严格精确对数似然
- 更像一个工程上可用的近似 surrogate

影响：

- RL 梯度方向可能仍然有用
- 但从概率建模上并不完全严谨

这是整个工程里最值得二次核查的地方之一。

## 2.3 论文说验证时 deterministic inference，代码在 selector 推理后又做随机增强

论文：

- `eta = 0` 做 deterministic inference

代码：

- 确实在 DDIM step 用了 `eta=0.0`
- 但 selector `forward_test_rl` 后面又调用 `add_mul_noise(diffusion_output)`

这意味着：

- 最终进入 scorer 的候选集仍然带随机性增强

影响：

- 如果不固定随机种子，推理结果可能有波动
- 这和“完全 deterministic inference”的论文表述并不完全一致

## 2.4 selector 默认不返回 trajectory，与官方评测接口不兼容

当前默认 `forward_test_rl` 返回：

- `reward_dict`

而官方 planner API 需要：

- `predictions["trajectory"]`

README 也明确提醒要手动取消注释返回 trajectory 的那行。

影响：

- 默认代码更偏向离线评测脚本
- 不是开箱即用的线上 planner 接口

这不是小问题，而是“接口层面未完全收口”的典型研究代码特征。

## 3. 代码中的硬编码与脆弱点

## 3.1 selector 训练 / 推理的 `num_groups` 是硬编码，不完全听配置

配置文件里：

- selector config `num_groups = 8`

但代码里：

- selector 训练 `num_groups = 2`
- selector 推理 `num_groups = 10`

这说明：

- `config.num_groups` 并没有统一驱动整个 selector 逻辑

影响：

- 改配置未必生效
- 实验可复现性会受影响

## 3.2 `topk=32`、`step_num=2/10` 等关键超参直接写死在函数里

例如：

- RL rollout `step_num = 10`
- 推理 `step_num = 2`
- selector top-k `topk = 32`

影响：

- 代码可调性差
- 很难做系统化 ablation

## 3.3 `CacheOnlyDataset` 的 metric cache 路径推导是字符串替换

它会把：

- `training_cache`

替换成：

- `train_pdm_cache`

并手动插入：

- `unknown`

影响：

- 强依赖特定目录命名
- 一旦路径组织方式变化，很容易 silent failure

## 4. 遗留接口、未使用模块和“看起来在，实际上没进主链”的部分

## 4.1 `ConditionalUnet1D` 被导入了，但 V2 主链没有真正用它

相关位置：

- `diffusiondrivev2_model_rl.py`
- `diffusiondrivev2_model_sel.py`
- `modules/conditional_unet1d.py`

观察：

- 被导入
- 但主模型实际 denoiser 走的是 `CustomTransformerDecoder`

结论：

- 这是典型遗留模块

## 4.2 `LossComputer` 在 V2 中被实例化，但主训练损失没真正依赖它

相关位置：

- `TrajectoryHead.__init__` 中有 `self.loss_computer = LossComputer(config)`

但：

- RL 版主损失走 `get_rlloss`
- selector 版主损失走 `_score_coarse/_score_fine_multi`

结论：

- 这是从原版 DiffusionDrive 继承下来的残留痕迹

## 4.3 `transfuser_loss.py` 在 V2 agent 中基本没进主链

虽然：

- RL / selector agent 都 import 了 `transfuser_loss`

但实际：

- RL agent 直接读取 `predictions['loss']`
- selector agent 直接读取 `predictions['loss']`

结论：

- 这是保留旧接口但不再是主路径

## 4.4 `compute_diversity` 只存在于 selector 模型里，主训练与推理未使用

说明：

- 这是分析 / 实验辅助函数，而不是核心逻辑

## 5. 工程上可疑或值得重点检查的实现

## 5.1 RL 训练没有显式 old policy / reference model

论文和 GRPO 语义上通常会让你以为存在：

- 当前 policy
- old/reference policy

但代码里实际上是：

1. 用当前模型 `no_grad` rollout
2. 再用同一个当前模型重算 log_prob

这意味着：

- 没有显式参数隔离的 old policy
- 也没有真正持久化 reference model

影响：

- 更像“同策略自采样 + detach rollout”
- 与标准 PPO/GRPO 的实现形态有差异

## 5.2 RL 版模型导入的是原版 `diffusiondrive` 包下的一些模块

例如：

- `diffusiondrive.transfuser_backbone`
- `diffusiondrive.modules.blocks`

而 selector 版多数用的是 `diffusiondrivev2` 自己的模块。

影响：

- 增加阅读成本
- 也增加未来重构时混淆依赖的风险

## 5.3 快速评测不是“真实在线推理”

`run_pdm_score_fast.py` 的本质是：

- cache feature
- cache metric
- 模型内部直接打 reward

它能快速复现实验指标，但不等于：

- 真正在线 planner 的调用流程

如果你以后做部署或仿真联调，这一点一定要分开。

---

## 6. 哪些配置看起来存在，但在 V2 主链里没有充分发挥作用

这些字段在 config 里存在，但在 V2 主训练主链里不是核心控制项：

- `trajectory_weight`
- `trajectory_cls_weight`
- `trajectory_reg_weight`
- `diff_loss_weight`
- `agent_class_weight`
- `agent_box_weight`
- `bev_semantic_weight`

原因：

- 这些权重更多服务于原版 DiffusionDrive 的 `transfuser_loss`
- V2 主训练走的是 RL / selector 自己的损失路径

这说明：

- 当前 config 有一部分是从原工程继承下来的
- 不要误以为这些权重一定在 V2 训练里直接生效

## 7. 二次开发最值得优先切入的 5 个方向

## 7.1 把 sampler / selector / reward bridge 彻底配置化

优先理由：

- 现在很多关键超参硬编码
- 做实验不方便

建议提炼为配置项：

- `rollout_step_num`
- `inference_step_num`
- `num_groups`
- `topk`
- `n_aug`
- `aug_std_range`

## 7.2 重写 `DDIMScheduler_with_logprob` 的概率建模

优先理由：

- 这是 RL 正确性的基础

建议方向：

- 推导乘性噪声对应的精确 log density
- 或改成更容易写出精确似然的参数化方式

## 7.3 把 official eval 和 fast eval 的接口统一

优先理由：

- 当前一个默认返回 `reward_dict`
- 一个接口要求 `trajectory`

建议做法：

- 统一返回结构，例如：
  - `trajectory`
  - `all_candidates`
  - `scores`
  - `reward_dict`

这样：

- 离线分析和官方评测都能走同一条主链

## 7.4 把生成器与 selector 的共享部分抽成公共模块

当前问题：

- RL 版和 selector 版有大段重复代码
- 但又有微妙差异

建议抽象：

- shared trajectory sampler
- shared anchor encoder
- shared scheduler wrapper

## 7.5 清理遗留模块与旧接口

建议优先清理：

- `ConditionalUnet1D`
- 未使用的 `LossComputer` 路径
- 旧 `transfuser_loss` 路径
- 不再使用的 imports

好处：

- 降低阅读门槛
- 降低维护成本

---

## 8. 如果你想把它迁移到自己的工程，最值得保留的设计思想

### 8.1 用 anchor 把多模态规划空间显式分块

这比让模型直接从纯噪声里“自己学会多模态”稳得多。

### 8.2 把轨迹点当成去地图上读上下文的 query

`GridSampleCrossBEVAttention` 非常适合规划问题。

### 8.3 先学生成，再学选择

把“造候选”和“挑候选”分开训练，很实用。

### 8.4 用闭环指标而不是纯 imitation target 当优化信号

这是从“像专家”转向“结果更好”的关键。

### 8.5 coarse-to-fine selector

大候选集先粗筛，小候选集再细排，是很强的工程模式。

---

## 9. 最后给你的学习建议

如果你现在准备自己二次开发，不建议一上来就改 backbone。  
最值得先改的是下面三处：

1. `TrajectoryHead.get_rlloss`
2. `DDIMScheduler_with_logprob.step`
3. `diffusiondrivev2_model_sel.py::forward_test_rl`

原因很简单：

- 这三处分别决定了
  - RL 是否合理
  - 采样概率是否自洽
  - 推理接口是否真正可部署

而这些正是这份工程最有研究价值、也最有工程价值的地方。
