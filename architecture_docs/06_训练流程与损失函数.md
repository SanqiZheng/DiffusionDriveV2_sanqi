# 06 训练流程与损失函数

## 1. 训练总览

DiffusionDriveV2 实际上有两套完全不同的训练流程：

1. Stage I：训练 RL 版生成器
2. Stage II：训练 selector

这两套流程虽然共用同一套感知 backbone 和 anchor diffusion 思路，但优化目标完全不同。

## 2. 一个 batch 是怎么流进系统的

统一入口：

- `navsim/planning/script/run_training.py`
- `navsim/planning/training/agent_lightning_module.py`

一个 batch 到 loss 的主路径：

```text
DataLoader
  -> AgentLightningModule._step
  -> agent.forward(features, targets, pdm_token_path, token)
  -> predictions
  -> agent.compute_loss(...)
  -> loss_dict["loss"]
```

如果 batch 来自 `CacheOnlyDataset`，`_step` 会得到：

- `features`
- `targets`
- `pdm_token_path`
- `token`

这就是为什么 v2 agent 的 `forward(...)` 多了：

- `metric_cache`
- `token`

## 3. Stage I：RL 生成器训练

文件：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_rl_agent.py`
- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py`

## 3.1 参数冻结策略

在 agent 初始化时：

- 非 `_trajectory_head` 参数全部 `requires_grad=False`
- 非 `_trajectory_head` 模块全部切到 `eval()`
- 只训练 `_trajectory_head`

这说明：

- RL 阶段不是从头训练整网
- 而是在 DiffusionDrive 预训练基础上，只微调轨迹生成头

## 3.2 `V2TransfuserModel.forward` 的双阶段前向

这一段很重要。

RL 版顶层 `forward` 里做了：

1. 先 `with torch.no_grad()` 调一次 `_trajectory_head(... old_pred=None ...)`
2. 得到 `old_pred`
3. 再调一次 `_trajectory_head(... old_pred=old_pred ...)`

它们分别代表：

- 第一次：无梯度 rollout 轨迹、算 reward 和 advantage
- 第二次：拿第一次 rollout 的链条重新算 RL loss

所以训练不是一次 forward 完成，而是：

- 先采样
- 再回放

## 3.3 `forward_train_rl`：先采样与算 reward

函数：

- `TrajectoryHead.forward_train_rl`

### 步骤 1：从 anchor 复制出组内样本

```python
plan_anchor = self.plan_anchor.unsqueeze(0).unsqueeze(0).repeat(bs, num_groups, 1, 1, 1)
plan_anchor = plan_anchor.view(bs, num_groups * self.ego_fut_mode, ...)
```

语义：

- `ego_fut_mode = 20` 是 20 个 anchor
- `num_groups` 是每个 anchor 复制多少条采样链

于是：

- 总候选数 `G_all = num_groups * 20`

默认 RL config：

- `num_groups = 4`

所以：

- 每个样本实际 rollout 80 条候选轨迹

### 步骤 2：加截断噪声

```python
diffusion_output = self.norm_odo(plan_anchor)
noise = torch.randn(...)
trunc_timesteps = 8
diffusion_output = self.diffusionrl_scheduler.add_noise(...)
```

含义：

- 从 anchor 附近的 truncated noisy trajectory 开始

### 步骤 3：逐步 denoise

每个 diffusion step 做：

1. 把当前 noisy traj 反归一化成真实坐标
2. 位置编码
3. 加 timestep embedding
4. 过 `diff_decoder`
5. 得到 `poses_reg`
6. 取 `poses_reg[..., :2]` 作为 `x_start`
7. 用 `diffusionrl_scheduler.step(...)` 更新到前一步

### 步骤 4：把最终 `xy` 变成 `xyyaw`

函数：

- `bezier_xyyaw`

作用：

- 在最前面补原点
- 通过 Bézier 一阶导数估 yaw

最终轨迹 shape：

- `(B, G_all, 8, 3)`

### 步骤 5：通过 PDM 计算 reward

函数：

- `get_pdm_score_para`

调用链：

- `TrajectoryHead.get_pdm_score_para`
- `_pdm_worker`
- `pdm_score_para`
- `PDMSimulator`
- `PDMScorer`

得到：

- `reward_group`
- `sub_rewards_group`

其中 `sub_rewards_group` 包含：

- `no_collision`
- `drivable_area`
- `progress`
- `ttc`
- `comfort`
- `dir_weighted`
- `final`

## 3.4 Intra-Anchor GRPO 在代码里怎么落地

最关键的一步是：

```python
reward_group = reward_group.view(bs, num_groups, self.ego_fut_mode)
mean_grouped_rewards = reward_group.mean(dim=1)
std_grouped_rewards = reward_group.std(dim=1)
advantages = (reward_group - mean_grouped_rewards.unsqueeze(1)) / (std_grouped_rewards.unsqueeze(1) + 1e-4)
```

要特别注意这个 shape：

- `reward_group.shape = (B, num_groups, 20)`

这里不是“20 组，每组 num_groups 个 anchor”，而是：

- 第二维 `num_groups`：同一个 anchor 的不同采样样本
- 第三维 `20`：anchor id

因此 `mean(dim=1)` 的含义正是：

- 在同一个 anchor 内，对不同样本做 group-relative baseline

这就是论文里的 `Intra-Anchor GRPO` 在代码里的直接落地。

## 3.5 Inter-Anchor Truncated GRPO 在代码里怎么落地

代码实现分两步。

### 第一步：只保留优于 GT 的正优势

```python
mask_positive = (reward_group > (reward_gt - 1e-6))
advantages = advantages.clamp(min=0) * mask_positive.float()
```

含义：

- 小于等于 GT 的样本，不给正向强化
- 负优势也直接截断到 0

### 第二步：对绝对失败样本给强惩罚

```python
if k == 'no_collision' or k == 'drivable_area':
    zero_mask = (v != 1)
    advantages = torch.where(zero_mask, torch.full_like(advantages, -1.0), advantages)
```

含义：

- 撞了或者出了 drivable area，就直接设成 `-1`

这就是论文的：

- “保留正向改进”
- “绝对失败强惩罚”

## 3.6 `get_rlloss`：真正回传的 RL loss

函数：

- `TrajectoryHead.get_rlloss`

输入来自：

- `old_pred['all_diffusion_output']`
- `old_pred['advantages']`

### 步骤 1：把 rollout 轨迹链拆成相邻状态对

```python
chains = old_diffusion_output[..., :-1]
chains_prev = old_diffusion_output[..., 1:]
```

### 步骤 2：重新逐步计算 log_prob

对每个 diffusion step：

1. 把 `chains[..., i]` 当成当前状态
2. 重新跑 decoder 得到 `x_start`
3. 调 `diffusionrl_scheduler.step(... prev_sample=chains_prev[..., i])`
4. 取回该 step 的 `log_prob`

### 步骤 3：形成 policy gradient proxy

```python
per_token_loss = -torch.exp(per_token_logps - per_token_logps.detach()) * advantages
```

这一写法的直觉是：

- 前向值约等于 `-advantages`
- 梯度只对 `log_prob` 保留

可以把它看成一种 REINFORCE 的稳定实现。

### 步骤 4：再加 imitation regularization

代码里的 IL loss：

```python
traj_l1 = F.l1_loss(poses_reg[..., :2], target_traj[..., :2], reduction='none')
```

它会对：

- 每个 diffusion step
- 每层 decoder 输出

都算一遍轨迹 L1，然后平均。

### 步骤 5：动态混合 RL 与 IL

```python
has_positive = (advantages > 0).any(dim=2).any(dim=1)
il_weight = torch.where(has_positive == 0, 1.0, 0.1)
loss = RL_loss + il_weight * IL_loss
```

解释：

- 如果一个 batch 里完全没有正优势样本，说明 RL 信号太弱
- 这时更依赖 imitation regularization
- 否则就以 RL 为主，只给 IL 一个较小权重

## 3.7 论文训练目标和代码训练目标的关系

### 论文说法

- `L = L_RL + λ L_IL`

### 代码实际

- `L_RL`：确实存在
- `L_IL`：代码里是多步轨迹 L1
- `λ`：不是固定超参数，而是动态 `0.1` 或 `1.0`

所以准确说：

论文思想被实现了，但实现形式更工程化。

## 4. Stage II：selector 训练

文件：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_sel_agent.py`
- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_sel.py`

## 4.1 参数冻结策略

selector agent 只训练这些前缀：

- `_trajectory_head.plan_anchor_scorer_encoder`
- `_trajectory_head.scorer_decoder`
- `_trajectory_head.NC_head`
- `_trajectory_head.DAC_head`
- `_trajectory_head.EP_head`
- `_trajectory_head.C_head`
- `_trajectory_head.TTC_head`
- `_trajectory_head.fine_*`

而：

- 生成器
- backbone
- diffusion decoder

都被冻结。

这完全符合论文“Stage II 训练 selector”的设定。

## 4.2 selector 训练的前半段：无梯度生成候选

`forward_train_rl` 一开始有：

```python
with torch.no_grad():
    ...
```

里面会：

1. 用 2-step diffusion 生成候选轨迹
2. 调 `add_mul_noise` 做额外 multiplicative augmentation
3. 调 `get_pdm_score_para` 拿每条候选的真实 PDM 子指标
4. 调 `get_vocab_pdm_subscores` 拼接 GTRS 词表候选

所以 selector 的训练数据不是固定候选集，而是：

- 模型生成候选
- augmented 候选
- 小部分外部词表候选

## 4.3 coarse scorer loss

函数：

- `_score_coarse`

预测头：

- `NC_score`
- `EP_score`
- `DAC_score`
- `TTC_score`
- `C_score`

损失组成：

1. `loss_nc`：BCE
2. `loss_ep`：BCE
3. `loss_dac`：BCE
4. `loss_ttc`：BCE
5. `loss_c`：BCE with mask
6. `loss_rank`：对 progress 的 pairwise margin ranking

最终：

```python
loss_coarse = loss_nc + loss_ep + loss_dac + loss_ttc + loss_c + 2 * loss_rank
```

## 4.4 fine scorer loss

函数：

- `_score_fine_multi`

做法和 coarse 类似，只是：

- 输入是 coarse top-k
- 有 3 层 fine scorer
- 每层都单独算一套子头
- 最后对 3 层 loss 取平均

## 4.5 selector 的 top-k 筛选

函数：

- `_select_topk`

流程：

1. 根据 coarse reward 取 `topk=32`
2. gather 对应的轨迹特征与轨迹点
3. 送入 fine scorer

这一步是论文 `coarse-to-fine selector` 的直接实现。

## 5. 所有 loss 项分别是什么意思

## 5.1 RL 版

### `loss`

- 最终训练损失
- `RL + dynamic IL`

### `reward`

- 只是日志项
- 表示当前 batch 的平均正样本质量

### `sub_rewards`

- 各个 PDM 子指标的均值

## 5.2 selector 版

### `coarse_loss_nc`

- 学习 no collision 子指标

### `coarse_loss_ep`

- 学习 progress 子指标

### `coarse_loss_dac`

- 学习 drivable area compliance

### `coarse_loss_ttc`

- 学习 TTC

### `coarse_loss_c`

- 学习 comfort

### `coarse_loss_rank`

- 让 progress 排序更合理

### `fine_loss_*`

- 对 top-k 候选做细排

## 6. 一个 batch 从 dataloader 到 loss 的完整路径

### RL 版

```text
batch
  -> features / targets / metric_cache_path / token
  -> backbone 编码场景
  -> diffusion 轨迹 rollout
  -> PDM reward
  -> advantage
  -> replay 同一扩散链
  -> RL logprob loss
  -> imitation trajectory L1
  -> final loss
```

### selector 版

```text
batch
  -> backbone 编码场景
  -> no_grad 生成候选轨迹
  -> PDM score 作为 selector supervision
  -> coarse scorer
  -> top-k
  -> fine scorer
  -> BCE + rank losses
  -> final loss
```

## 7. 论文与代码在损失上的几个关键差异

### 7.1 论文里的 BC / IL 在代码中是 trajectory L1

不是一个抽象的 KL 或 BCE regularizer，而是：

- 多步、多层 decoder 输出对 GT 轨迹的 L1

### 7.2 论文里的 selector “score learning” 在代码中被拆成了多头监督

代码不是直接回归总分，而是先学：

- NC / DAC / TTC / EP / C

再把它们组合。

### 7.3 reward 是通过 PDM 闭环计算出来的，不是直接监督

这一点决定了整个训练很“重”：

- reward 依赖 simulator
- reward 依赖 metric cache
- reward 依赖多进程

## 8. 一句话总结训练逻辑

Stage I 训练的是“如何生成一组更优的多模态轨迹”，Stage II 训练的是“如何从这些轨迹里挑最好的”。  
前者的核心是 `reward -> advantage -> logprob`，后者的核心是 `PDM 子指标监督 -> coarse-to-fine ranking`。
