# 08 从基础理论到项目算法的数学推导

## 1. 这一章要解决什么问题

这章的目标不是把论文公式简单抄一遍，而是把下面这条链讲清楚：

```text
最基础的 diffusion / DDIM
  -> anchor-conditioned truncated diffusion
  -> 代码里到底预测什么
  -> denoising chain 为什么可以看成 policy
  -> group relative advantage 怎么定义
  -> 代码里的 loss 到底在优化什么
```

如果你以前没系统学过 diffusion，这一章建议慢慢读。

## 2. 从最基础开始：什么是轨迹生成问题

在这个项目里，我们要生成的是未来 ego 轨迹：

```math
\tau = \{(x_n, y_n, \theta_n)\}_{n=1}^{N_f}
```

当前工程默认：

- `N_f = 8`
- 时间跨度 4 秒
- 采样间隔 0.5 秒

代码对象对应：

- 数学里的 `\tau`
- 对应代码里的 `targets["trajectory"]`
- shape 是 `(8, 3)` 或 batched `(B, 8, 3)`

## 3. 普通 diffusion 的最小背景

标准 diffusion 的基本思想是：

1. 前向过程：逐步加噪，把真实样本变成噪声
2. 反向过程：学习如何一步步去噪，把噪声变回样本

如果把真实样本记成 `x_0`，第 `t` 步 noisy sample 记成 `x_t`，则常见前向过程写成：

```math
q(x_t \mid x_0) = \mathcal{N}\left(\sqrt{\bar\alpha_t}x_0,\ (1-\bar\alpha_t)I\right)
```

直觉上：

- `t` 越大，`x_t` 越像纯噪声
- 模型要学的是怎样从 `x_t` 恢复到更干净的状态

## 4. DDIM / DDPM 在这个项目里的位置

本项目使用的是 `diffusers` 里的 `DDIMScheduler`。

在 DDIM 里，模型通常有三种常见预测目标：

1. 预测噪声 `epsilon`
2. 预测样本 `x_0`
3. 预测 velocity `v`

而当前代码明确设置：

```python
prediction_type="sample"
```

所以数学上这里学的是：

```math
f_\theta(x_t, t, c) \approx x_0
```

其中：

- `x_t` 是当前 noisy trajectory
- `t` 是 diffusion step
- `c` 是条件信息，比如地图、周围车、ego 状态

代码对象对应：

| 数学对象 | 代码对象 |
| --- | --- |
| `x_t` | `diffusion_output` |
| `f_\theta(x_t, t, c)` | `poses_reg[..., :2]` 经过 `norm_odo` 后的 `x_start` |
| `t` | `timesteps` |
| `c` | `bev_feature`, `agents_query`, `ego_query`, `status_encoding` 等 |

## 5. 为什么这不是普通 vanilla diffusion

普通 diffusion 常常从纯高斯噪声开始。  
DiffusionDriveV2 不是这样。

它先准备 20 个轨迹 anchor：

```math
\{a^k\}_{k=1}^{N_{\text{anchor}}}, \quad N_{\text{anchor}}=20
```

每个 anchor 表示一个驾驶意图子空间，例如：

- 直行
- 左转
- 右转
- 变道

于是模型不是从 `z ~ N(0, I)` 直接生成，而是从：

```math
x_t^k \approx \sqrt{\bar\alpha_t} a^k + \sqrt{1-\bar\alpha_t}\,\epsilon
```

开始。

这就是论文里的：

- `Gaussian Mixture Model prior`
- `anchor-based truncated diffusion`

代码对象对应：

- `plan_anchor`
- `self.diffusion_scheduler.add_noise(original_samples=norm_odo(plan_anchor), ...)`

## 6. 为什么叫 truncated diffusion

因为它不是把时间步从 1000 一路还原到底，而是：

- 先用完整 1000-step scheduler 的噪声定义
- 但实际只从一个较浅的 `trunc_timesteps=8` 开始
- 再用很少的推理步数走完

例如代码里：

- RL 训练 rollout：`step_num = 10`
- 推理：`step_num = 2`

直觉上：

- 模型不是在特别远的纯噪声空间里挣扎
- 而是在“anchor 附近的局部扰动”里修轨迹

这能显著减少采样成本。

## 7. 从轨迹生成到项目的具体数学对象

### 7.1 代码里真正被 diffusion 建模的是谁

代码中被建模的核心对象是：

```math
x_t \in \mathbb{R}^{8 \times 2}
```

也就是未来 8 个 `(x, y)` 点。

为什么不是 `(x, y, \theta)`：

- 生成阶段主输出只用 `poses_reg[..., :2]`
- 最后的 yaw 是后处理 `bezier_xyyaw` 推出来的

所以更精确地说：

- diffusion 学的是未来 8 个二维轨迹点
- heading 不是 diffusion 直接预测的主对象

## 7.2 归一化空间

代码里先做：

```math
\tilde x = \text{norm\_odo}(x)
```

大致相当于：

```math
\tilde x = (x / 50,\ y / 20)
```

所以 diffusion 工作的实际空间是：

- 一个归一化轨迹空间

代码对象对应：

- `norm_odo`
- `denorm_odo`

## 8. 反向过程在代码里怎么体现

在每个 step，模型先根据当前 noisy trajectory 预测更干净的轨迹：

```math
\hat x_0 = f_\theta(x_t, t, c)
```

代码里：

```python
poses_reg = poses_reg_list[-1]
x_start = self.norm_odo(poses_reg[..., :2])
```

随后 scheduler 用 DDIM 公式把 `x_t` 更新到 `x_{t-1}`：

```math
x_{t-1} = \sqrt{\bar\alpha_{t-1}}\hat x_0 + \sqrt{1-\bar\alpha_{t-1}-\sigma_t^2}\,\hat\epsilon + \sigma_t z
```

在代码的 `DDIMScheduler_with_logprob.step` 中，对应的是：

- `pred_original_sample`
- `pred_epsilon`
- `prev_sample_mean`
- `prev_sample`

## 9. 从 diffusion 到 policy：为什么 denoising chain 可以做 RL

论文的关键想法是：

把每个 denoising step 看成一次 action。

于是我们可以把整个生成链写成一个序列决策问题：

- state：当前 `x_t`
- action：生成 `x_{t-1}`
- policy：`\pi_\theta(x_{t-1} \mid x_t, c)`
- trajectory-level reward：最终生成出的整条轨迹经过闭环评测得到的分数

如果把第 `k` 个 anchor 下第 `i` 条采样链记成一条 rollout，那么它的总目标可以写成：

```math
J(\pi_\theta) = \mathbb{E}_{\pi_\theta}\left[R(x_0)\right]
```

这里的 `R(x_0)` 在本项目里不是简单函数，而是：

```text
trajectory
  -> PDMSimulator
  -> PDMScorer
  -> final reward
```

## 10. 最基本的 policy gradient 形式

REINFORCE 告诉我们：

```math
\nabla_\theta J(\pi_\theta)
=
\mathbb{E}\left[
\sum_t \nabla_\theta \log \pi_\theta(a_t \mid s_t)\, A_t
\right]
```

映射到本项目里：

- `s_t` 对应 `x_t`
- `a_t` 对应一次 denoising update
- `A_t` 对应 advantage

代码对象对应：

| 数学对象 | 代码对象 |
| --- | --- |
| `s_t` | `diffusion_output` |
| `a_t` | `prev_sample` |
| `\log \pi_\theta(a_t \mid s_t)` | `log_prob` |
| `A_t` | `advantages` |

## 11. Intra-Anchor GRPO 的数学直觉

如果直接把所有样本混在一起做 group-relative advantage，会有一个问题：

- 不同 anchor 表示不同驾驶意图
- 它们不应该简单放进同一组做相对优劣比较

所以论文提出：

对每个 anchor 单独构造一个组，组内做相对优势估计。

如果第 `k` 个 anchor 下采样了 `G` 条轨迹，记 reward 为：

```math
r^{k,1}, r^{k,2}, \dots, r^{k,G}
```

则 group-relative advantage 可写成：

```math
A^{k,i}
=
\frac{
r^{k,i} - \text{mean}(r^{k,1}, \dots, r^{k,G})
}{
\text{std}(r^{k,1}, \dots, r^{k,G})
}
```

代码里正对应：

```python
reward_group = reward_group.view(bs, num_groups, ego_fut_mode)
mean_grouped_rewards = reward_group.mean(dim=1)
std_grouped_rewards = reward_group.std(dim=1)
advantages = ...
```

其中：

- `num_groups` 就是每个 anchor 的样本数 `G`
- `ego_fut_mode=20` 就是 anchor id

## 12. Inter-Anchor Truncated GRPO 的数学直觉

光有组内相对优势还不够，因为：

- 某个 anchor 组里“最好”的样本
- 在全局上可能仍然很差

论文的做法是把 advantage 再做一次截断：

```math
A_{\text{trunc}}^{k,i}
=
\begin{cases}
-1, & \text{if absolute failure} \\
\max(0, A^{k,i}), & \text{otherwise}
\end{cases}
```

代码实现稍微更具体：

1. 先只保留优于 GT 的正优势
2. 再对 collision / drivable-area failure 直接设成 `-1`

所以代码版实际上是：

```math
A_{\text{code}}^{k,i}

=
\mathbf{1}[r^{k,i} > r_{\text{gt}}]\cdot \max(0, A^{k,i})
```

再叠加失败掩码：

```math
A_{\text{code}}^{k,i} = -1 \quad \text{if collision or off-road}
```

## 13. RL loss 到代码形式的推导

理想化写法可以写成：

```math
L_{\text{RL}}
=
-
\sum_t \log \pi_\theta(x_{t-1} \mid x_t, c)\, A_t
```

代码没有直接这么写，而是用了：

```python
per_token_loss = -torch.exp(per_token_logps - per_token_logps.detach()) * advantages
```

为什么这么写？

因为：

```math
\exp(\log p - \text{stopgrad}(\log p))
```

前向值等于 1，但反向保留了关于 `\log p` 的梯度。  
于是它实现的是一种：

- 数值更稳定
- 梯度方向等价于 policy gradient

的写法。

## 14. 论文里的 `L = L_RL + \lambda L_IL` 到代码怎么变形

论文表达：

```math
L = L_{\text{RL}} + \lambda L_{\text{IL}}
```

代码表达：

```python
loss_b = RL_loss_b + il_weight * IL_loss_b
```

这里：

- `RL_loss_b` 是 batch 内每个样本的 RL 损失
- `IL_loss_b` 是多 step、多层 decoder 输出对 GT 轨迹的 L1
- `il_weight` 是动态的：
  - 有正优势样本时用 `0.1`
  - 否则用 `1.0`

所以代码不是严格固定 `\lambda`，而是：

```math
\lambda = \lambda(\text{batch})
```

## 15. selector 的数学对象

selector 的任务可以理解成：

输入一组候选轨迹：

```math
\{\tau_1, \tau_2, \dots, \tau_M\}
```

学习一个评分函数：

```math
s_\phi(\tau_m, c)
```

项目里它不是直接学一个总分，而是先学多个子分量：

- `s_{\text{NC}}`
- `s_{\text{DAC}}`
- `s_{\text{TTC}}`
- `s_{\text{EP}}`
- `s_{\text{C}}`

再组合成总 reward：

```math
s_{\text{final}}
=
\sigma(s_{\text{NC}})\sigma(s_{\text{DAC}})
\cdot
\frac{
5\sigma(s_{\text{TTC}})
+
5\sigma(s_{\text{EP}})
+
2\sigma(s_{\text{C}})
}{12}
```

这对应代码里的：

- `final_coarse_reward`
- `final_fine_reward`

## 16. 为什么 selector 还需要 rank loss

如果只用 BCE 学各子指标，模型更像在学“是否满足阈值”。  
但规划里还有大量连续细差别，比如：

- 都不撞，但谁更顺
- 都可行，但谁 progress 更高

所以又加入：

```math
L_{\text{rank}}
=
\max(0,\ -\text{sign}(s_i - s_j)(\hat s_i - \hat s_j) + m)
```

代码里体现在：

- `_score_coarse`
- `_score_fine_multi`

里的 `MarginRankingLoss(margin=0.05)`

## 17. 数学对象与代码对象的一一对应

| 数学对象 | 含义 | 代码对象 |
| --- | --- | --- |
| `a^k` | 第 `k` 个 anchor | `plan_anchor[k]` |
| `x_t` | 第 `t` 步 noisy trajectory | `diffusion_output` |
| `\hat x_0` | 预测的 clean sample | `x_start` |
| `c` | 条件信息 | `bev_feature`, `agents_query`, `ego_query`, `status_encoding` |
| `\pi_\theta` | denoising policy | `DDIMScheduler_with_logprob.step` 隐式定义的 step policy |
| `R(x_0)` | 最终 reward | `get_pdm_score_para` 输出 |
| `A^{k,i}` | 组内相对优势 | `advantages` |
| `A_{\text{trunc}}^{k,i}` | 截断后的优势 | `advantages.clamp(min=0)` 与 failure penalty 后结果 |
| `L_{\text{RL}}` | RL loss | `per_token_loss` 聚合后结果 |
| `L_{\text{IL}}` | imitation regularization | `IL_loss_b` |
| `s_\phi` | selector score | `NC_head`, `EP_head`, ... 输出 |

## 18. 论文与代码在数学上最值得警惕的差异

### 18.1 代码的 `log_prob` 不是乘性噪声下的严格精确对数似然

代码采样时用了：

```python
prev_sample = prev_sample_mean * variance_noise_mul + ...
```

但 `log_prob` 仍然用的是类似各向同性高斯残差形式：

```python
-((prev_sample - prev_sample_mean)^2) / (2 * std^2)
```

这从严格概率论上看并不完全匹配乘性噪声分布。  
也就是说：

- 论文的 policy 公式是一个理想化高斯策略
- 代码实现是一个工程上的近似 RL surrogate

### 18.2 论文的固定 `\lambda`，代码是动态 `il_weight`

这是数学目标到工程实现的明显改写。

## 19. 最后用一句最朴素的话收束

这个项目的数学本质可以概括成：

先在 anchor 条件下用截断 diffusion 生成多模态轨迹，再把 denoising chain 当作 policy rollout，用闭环规划 reward 去优化它；由于不同 anchor 天然代表不同驾驶意图，所以 advantage 只能在同 anchor 内相对比较，而跨 anchor 只做截断式全局约束。
