# 02 论文到代码的映射关系

## 1. 总览

先给出一句总判断：

当前仓库确实实现了论文所说的“两阶段框架”：

1. `DiffusionDriveV2 RL`：训练扩散生成器
2. `DiffusionDriveV2 Selector`：冻结生成器，单独训练模式选择器

但同时，代码也存在一些明显的工程化改写、简化和不完全一致之处：

- 论文里很多公式写得更干净
- 代码里为了落地训练，加入了不少硬编码、缓存依赖、评测捷径和遗留接口

所以学习时一定要分清：

- 论文是“方法设计”
- 代码是“能跑起来的工程版本”

## 2. 论文核心概念到源码位置的主映射

### 2.1 Anchor / GMM 先验

论文含义：

- 用一组预聚类的 anchor 来表示不同驾驶意图
- 扩散从这些 anchor 的邻域开始，而不是从纯高斯白噪声开始

代码位置：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py`
  - `TrajectoryHead.__init__`
  - `self.plan_anchor = nn.Parameter(torch.tensor(plan_anchor), requires_grad=False)`
- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_sel.py`
  - `TrajectoryHead.__init__`
- 数据文件
  - `kmeans_navsim_traj_20.npy`

代码中的真实语义：

- `plan_anchor.shape = (20, 8, 2)`
- 20 表示 20 个 anchor / mode
- 每个 anchor 是未来 8 个时间点的 `(x, y)` 轨迹

### 2.2 Truncated diffusion

论文含义：

- 做少量 denoising steps，而不是标准长链 diffusion

代码位置：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py`
  - `TrajectoryHead.forward_train_rl`
  - `TrajectoryHead.forward_test_rl`
- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_sel.py`
  - `TrajectoryHead.forward_train_rl`
  - `TrajectoryHead.forward_test_rl`

对应实现：

- `DDIMScheduler(num_train_timesteps=1000, prediction_type="sample")`
- 训练和推理都从 `trunc_timesteps = 8` 开始
- RL 版训练里 `step_num = 10`
- RL 版推理里 `step_num = 2`
- selector 的生成阶段也只走 `step_num = 2`

结论：

- 论文说的 truncated diffusion 在代码里是真实落地的
- 但“真正 denoise 几步”是手工设置的，不是统一从配置里驱动

### 2.3 模型预测对象是什么

论文层面：

- diffusion 生成轨迹

代码层面最关键的判断：

- 这个项目学习的是 `x_start`
- 不是 `noise epsilon`
- 也不是 `velocity`

证据：

- `DDIMScheduler(... prediction_type="sample")`
- `x_start = poses_reg[..., :2]`
- 后续 `scheduler.step(model_output=x_start, sample=diffusion_output, ...)`

这意味着：

- 模型直接预测去噪后的轨迹样本
- 样本所在空间是“归一化后的轨迹坐标空间”

### 2.4 RL 目标 / GRPO

论文含义：

- 把 denoising chain 看成 MDP
- 用 Intra-Anchor GRPO 做组内相对优势估计
- 用 Inter-Anchor Truncated GRPO 提供全局约束

代码位置：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py`
  - `TrajectoryHead.forward_train_rl`
  - `TrajectoryHead.get_rlloss`

关键变量映射：

| 论文概念 | 代码变量 / 位置 | 说明 |
| --- | --- | --- |
| trajectory group per anchor | `reward_group.view(bs, num_groups, self.ego_fut_mode)` | 第一维 `num_groups` 是每个 anchor 的采样组数，第二维 `ego_fut_mode=20` 是 anchor id |
| group relative advantage | `advantages = (reward_group - mean_grouped_rewards.unsqueeze(1)) / (std_grouped_rewards.unsqueeze(1) + 1e-4)` | 先在同 anchor 的不同样本间做标准化 |
| truncate negative advantage | `advantages.clamp(min=0)` | 负优势截断为 0 |
| absolute failure penalty | 对 `no_collision` / `drivable_area` 非 1 的样本置 `-1.0` | 对失败样本强惩罚 |
| policy gradient term | `per_token_loss = -torch.exp(per_token_logps - per_token_logps.detach()) * advantages` | 用 detach 技巧保留 REINFORCE 梯度 |
| imitation regularization | `IL_loss_b` | 实际代码用的是轨迹 L1，不是论文写法中的抽象 `LIL` |

### 2.5 Scale-adaptive multiplicative noise

论文含义：

- 为了探索，噪声不能简单逐点相加
- 需要保留轨迹的整体平滑性

代码位置：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py`
  - `DDIMScheduler_with_logprob.step`
- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_sel.py`
  - `DDIMScheduler_with_logprob.step`
  - `add_mul_noise`

实现方式：

- 不是给每个轨迹点独立加 `(dx, dy)` 噪声
- 而是给整条轨迹乘上两个共享随机量：
  - 一个 longitudinal
  - 一个 lateral

换句话说，代码的探索更像：

- 整条轨迹整体拉伸 / 压缩 / 偏移

而不是：

- 把每个点抖成锯齿线

### 2.6 Mode selector

论文含义：

- coarse-to-fine selector
- BCE + Margin-Rank

代码位置：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_sel.py`
  - `TrajectoryHead._score_coarse`
  - `TrajectoryHead._score_fine_multi`
  - `TrajectoryHead._select_topk`
  - `TrajectoryHead.forward_train_rl`
  - `TrajectoryHead.forward_test_rl`

对应模块：

- `plan_anchor_scorer_encoder`
- `scorer_decoder`
- `NC_head`, `EP_head`, `DAC_head`, `TTC_head`, `C_head`
- `fine_scorer_decoder`
- `fine_NC_head`, `fine_EP_head`, `fine_DAC_head`, `fine_TTC_head`, `fine_C_head`

代码真实做法：

- selector 不是直接回归一个总分
- 而是先预测多个子指标头
- 再把这些子指标组合成总 reward

这比论文摘要里的“selector predict score”更细。

## 3. 训练入口、推理入口分别在哪里

### 3.1 训练入口

统一训练入口：

- `navsim/planning/script/run_training.py`

组装顺序：

1. Hydra 读取 `default_training.yaml`
2. `instantiate(cfg.agent)` 实例化 agent
3. `AgentLightningModule(agent=agent)` 包装成 LightningModule
4. `Dataset` / `CacheOnlyDataset` 构建数据集
5. `trainer.fit(...)`

具体 agent 配置：

- RL 生成器训练
  - `navsim/planning/script/config/common/agent/diffusiondrivev2_rl_agent.yaml`
- selector 训练
  - `navsim/planning/script/config/common/agent/diffusiondrivev2_sel_agent.yaml`

### 3.2 推理 / 采样入口

有两个不同层面的“推理”：

#### 离线快速评测入口

- `navsim/planning/script/run_pdm_score_fast.py`

特点：

- 加载缓存特征与 metric cache
- 走 `trainer.validate(...)`
- 在 `forward + compute_loss` 里直接计算 reward 指标
- 这是“离线带 oracle metric 的快速评测”

#### 官方 planner 评测入口

- `navsim/planning/script/run_pdm_score.py`

特点：

- 通过 `AbstractAgent.compute_trajectory(...)` 调 planner
- 需要 agent 的 `forward` 最终返回 `trajectory`

而当前 selector 模型默认没有返回 `trajectory`，只返回 `reward_dict`。  
这也是 README 里强调“要手动取消注释某一行”的原因。

## 7. 论文中的 loss / objective 在代码中的实现

### 7.1 生成器训练

论文：

- `L = L_RL + λ L_IL`

代码：

- RL loss 在 `get_rlloss`
- IL loss 是所有 decoder step 的轨迹 L1 累积平均
- 最终 `loss = RL_loss + il_weight * IL_loss`

注意：

- 论文写法更抽象
- 代码里 `il_weight` 不是固定常数
- 如果 batch 内没有正优势样本，权重会退回到 `1.0`
- 否则用 `0.1`

### 7.2 selector 训练

论文：

- BCE + Margin-Rank

代码：

- `_score_coarse`：多个二值子指标头的 BCE + progress rank loss
- `_score_fine_multi`：fine stage 的同类损失

## 8. 论文中的 sampling / decoding / solving 在代码中的实现

### 8.1 diffusion sampling

- `DDIMScheduler`
- `DDIMScheduler_with_logprob`

### 8.2 denoising network

- `CustomTransformerDecoder`
- `CustomTransformerDecoderLayer`

### 8.3 closed-loop reward solving

- `pdm_score_para`
- `PDMSimulator`
- `PDMScorer`

这里要特别注意：

论文里的“reward”不是模型内部自己算的 differentiable reward，  
而是通过 PDM 模拟器和 scorer 做出来的闭环指标。

## 9. 哪些地方论文与代码完全一致

一致度较高的部分：

1. anchor-based truncated diffusion 的整体结构
2. 两阶段训练：先 RL 生成器，再训练 selector
3. multiplicative exploration noise 的思想
4. Intra-Anchor 与 Inter-Anchor 的设计方向
5. coarse-to-fine selector
6. 使用 PDM 系列指标作为奖励 / 评估基准

## 10. 哪些地方代码做了工程化改动或与论文不完全一致

### 10.1 论文的 `LIL`，代码实际是动态加权的轨迹 L1

论文写的是：

- `L = L_RL + λ L_IL`

代码实际是：

- `IL_loss_b` 由多步 decoder 的轨迹 L1 组成
- `λ` 不是常数，而是根据 batch 中是否有正优势样本动态切换

### 10.2 论文把 policy 写成高斯，代码实现了乘性噪声版近似

论文公式更像标准高斯策略。  
代码实际采样时用的是乘性噪声，并额外手写了 `log_prob`。

因此：

- 思想一致：都把 denoising step 当 policy step
- 数学实现并不完全等价

### 10.3 selector 在代码里比论文摘要描述得更细

论文说 mode selector。  
代码实际是：

- 5 个 coarse 子头
- 5 个 fine 子头
- coarse-to-fine top-k
- 加 GTRS 词表增强

### 10.4 推理接口存在工程妥协

论文语义上 selector 最终应输出轨迹。  
代码默认为了快速评测，返回的是 `reward_dict`，而不是最终 `trajectory`。

## 11. 结论

如果把论文到代码的映射压缩成一句话：

DiffusionDriveV2 的论文创新点在代码里主要落在 `TrajectoryHead` 这一层，尤其是 RL 版中的 `forward_train_rl + get_rlloss`，以及 selector 版中的 `forward_train_rl + _score_coarse + _score_fine_multi`；而感知 backbone、数据构建、PDM 评测接口则是把这些创新真正接进完整工程的骨架。
