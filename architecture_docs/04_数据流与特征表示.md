# 04 数据流与特征表示

## 1. 原始数据到模型输入的完整链路

先看主链路：

```text
原始 log pkl
  -> SceneLoader.filter_scenes
  -> Scene / AgentInput
  -> TransfuserFeatureBuilder.compute_features
  -> TransfuserTargetBuilder.compute_targets
  -> Dataset / CacheOnlyDataset
  -> DataLoader
  -> agent.forward(features, targets, metric_cache, token)
```

涉及的关键文件：

- `navsim/common/dataloader.py`
- `navsim/common/dataclasses.py`
- `navsim/agents/diffusiondrivev2/transfuser_features.py`
- `navsim/planning/training/dataset.py`

## 2. 样本容器是什么

## 2.1 SceneFilter

文件：

- `navsim/common/dataclasses.py`

作用：

- 定义如何从长日志切片成训练样本

当前常见设置：

- `num_history_frames = 4`
- `num_future_frames = 10`
- `frame_interval = 1`

这意味着一个样本由：

- 4 帧历史
- 10 帧未来

组成。

## 2.2 Scene

文件：

- `navsim/common/dataclasses.py`

作用：

- 保存带有未来 GT 的完整样本

你会用到的方法：

- `scene.get_future_trajectory(...)`
- `scene.frames[...]`
- `scene.map_api`

## 2.3 AgentInput

文件：

- `navsim/common/dataclasses.py`

作用：

- 保存只给 agent 看的输入侧信息
- 包括历史 ego status、camera、lidar

从 `Scene` 到 `AgentInput` 的关系可以理解成：

- `Scene` 是“监督学习完整样本”
- `AgentInput` 是“推理时可见输入”

## 3. SceneLoader 是怎么取样本的

文件：

- `navsim/common/dataloader.py`

核心过程：

1. `filter_scenes(...)` 读取每个 log pickle
2. 按 `SceneFilter.num_frames` 滑窗切片
3. 根据 `has_route`、`tokens`、`log_names` 做过滤
4. 用当前历史帧最后一个 token 作为样本 id

这意味着：

- 一个样本本质上是连续帧切片
- token 对应的是“历史最后一帧”的标识

## 4. 传感器如何变成特征

文件：

- `navsim/agents/diffusiondrivev2/transfuser_features.py`

核心类：

- `TransfuserFeatureBuilder`

### 4.1 `camera_feature`

函数：

- `TransfuserFeatureBuilder._get_camera_feature`

处理过程：

1. 取当前时刻的三路相机：
   - `cam_l0`
   - `cam_f0`
   - `cam_r0`
2. 左右图先裁掉两侧
3. 三张图横向拼接
4. resize 到 `1024 x 256`
5. `transforms.ToTensor()`

最终张量：

- shape：`(3, 256, 1024)`
- 语义：三路前向视角拼接后的 RGB 图

### 4.2 `lidar_feature`

函数：

- `TransfuserFeatureBuilder._get_lidar_feature`

处理过程：

1. 取最后一帧 LiDAR 点云 `lidar_pc`
2. 只保留 `(x, y, z)`
3. 去掉高于 `max_height_lidar` 的点
4. 按高度阈值切成 above / below
5. 用 `np.histogramdd` 投影到二维栅格
6. 归一化为直方图强度

默认配置下：

- `use_ground_plane = False`
- 所以只保留 `above_features`

最终张量：

- shape：`(1, 256, 256)`
- 语义：单通道 LiDAR BEV histogram

### 4.3 `status_feature`

函数：

- `TransfuserFeatureBuilder.compute_features`

拼接项：

- `driving_command`：4 维
- `ego_velocity`：2 维
- `ego_acceleration`：2 维

最终张量：

- shape：`(8,)`
- 语义：ego 当前高层驾驶命令 + 速度 + 加速度

## 5. 目标如何构造

核心类：

- `TransfuserTargetBuilder`

### 5.1 `trajectory`

函数：

- `TransfuserTargetBuilder.compute_targets`

来源：

- `scene.get_future_trajectory(num_trajectory_frames=self._config.trajectory_sampling.num_poses)`

默认 `TrajectorySampling`：

- `time_horizon = 4`
- `interval_length = 0.5`

因此：

- `num_poses = 8`

最终张量：

- shape：`(8, 3)`
- 语义：未来 4 秒、每 0.5 秒一个点的 `(x, y, heading)`

### 5.2 `agent_states`

函数：

- `TransfuserTargetBuilder._compute_agent_targets`

处理逻辑：

1. 遍历当前历史最后一帧 annotation
2. 只保留 `vehicle`
3. 只保留落在 LiDAR 范围内的车辆
4. 取最近的最多 `num_bounding_boxes=30` 个

最终张量：

- shape：`(30, 5)`
- 语义：每个框的 `(x, y, heading, length, width)`

### 5.3 `agent_labels`

最终张量：

- shape：`(30,)`
- 语义：哪些 agent slot 有效

### 5.4 `bev_semantic_map`

函数：

- `TransfuserTargetBuilder._compute_bev_semantic_map`

来源：

- 地图 polygon / linestring
- annotation boxes

最终张量：

- shape：`(128, 256)`
- 语义：BEV 语义标签图

为什么是 `(128, 256)`：

- `bev_pixel_height = lidar_resolution_height // 2 = 128`
- `bev_pixel_width = 256`

## 6. 训练样本缓存格式

文件：

- `navsim/planning/training/dataset.py`

缓存方式：

- 每个 token 一个目录
- 每个 builder 一个 `.gz` 文件
- 内容是 pickle 后的 tensor dict

例如一个 token 目录里通常有：

- `transfuser_feature.gz`
- `transfuser_target.gz`

## 7. Dataset 如何返回 batch

### 7.1 常规训练

`Dataset.__getitem__` 返回：

- `(features, targets)`

### 7.2 使用训练缓存

`CacheOnlyDataset.__getitem__` 返回：

- `(features, targets, pdm_token_path, token)`

这里额外多出来两个对象：

- `pdm_token_path`
  - 指向训练时 reward 计算要用的 metric cache
- `token`
  - selector 训练时还会拿它去查 GTRS vocab 对应的分数表

### 7.3 快速评测缓存

`CacheOnlyDatasetTest.__getitem__` 返回：

- `(features, targets, metric_cache_path, token)`

## 8. 关键张量 shape 总表

下面这张表是学习时最值得记住的：

| 名称 | shape | 语义 |
| --- | --- | --- |
| `camera_feature` | `(B, 3, 256, 1024)` | 三路前向相机拼接图 |
| `lidar_feature` | `(B, 1, 256, 256)` | 单通道 LiDAR BEV 栅格 |
| `status_feature` | `(B, 8)` | command + velocity + acceleration |
| `trajectory` | `(B, 8, 3)` | GT 未来轨迹 |
| `agent_states` | `(B, 30, 5)` | GT 周围车辆框 |
| `agent_labels` | `(B, 30)` | 有效框 mask |
| `bev_semantic_map` | `(B, 128, 256)` | GT BEV 语义图 |
| `plan_anchor` | `(20, 8, 2)` | 20 个 anchor 轨迹 |
| `diffusion_output` | `(B, G_all, 8, 2)` 或 `(B, G_all, 8, 3)` | 当前候选轨迹 |

## 9. 条件输入是在哪里注入模型的

从数据流角度看，条件信息分三类注入：

### 9.1 视觉与 LiDAR 条件

先进入：

- `TransfuserBackbone`

输出：

- BEV memory
- fused features

### 9.2 ego 状态条件

在：

- `V2TransfuserModel._status_encoding`

被编码成：

- `status_encoding`

再拼进：

- `keyval = torch.concatenate([bev_feature, status_encoding[:, None]], dim=1)`

### 9.3 轨迹级条件

在扩散 decoder 内部通过：

- `GridSampleCrossBEVAttention`
- `cross_agent_attention`
- `cross_ego_attention`
- `time_mlp`

逐层注入。

## 10. 评测相关的数据流

DiffusionDriveV2 的训练不是只靠 GT target，它还需要 `metric_cache`。

对应数据流：

```text
metric cache
  -> CacheOnlyDataset / CacheOnlyDatasetTest
  -> model forward
  -> get_pdm_score_para
  -> pdm_score_para
  -> PDMSimulator / PDMScorer
  -> reward / metrics
```

这意味着：

- 没有 metric cache，就很难复现实验中的 RL reward
- selector 的快速评测也依赖 metric cache

## 11. 数据链路里值得注意的工程细节

### 11.1 训练 cache 到 PDM cache 的路径映射是硬编码字符串替换

`CacheOnlyDataset._load_scene_with_token` 里会：

- 把 `training_cache` 替换成 `train_pdm_cache`
- 然后手工插入 `unknown`

这说明：

- 当前工程对缓存目录结构有很强假设
- 如果你改缓存路径命名，可能直接失效

### 11.2 selector 训练还依赖 GTRS 词表数据

相关文件：

- `gtrs_traj/16384.npy`
- `gtrs_traj/navtrain_16384.pkl`（README 要单独下载）

这部分不属于 NAVSIM 原始数据，而是 selector 额外增强数据。

## 12. 一句话总结数据流

这个项目的数据流不是“图像 -> 模型”这么简单，而是：

`原始场景切片 -> 传感器与地图特征 -> 缓存 -> batch -> 扩散候选轨迹 -> PDM 闭环评分 -> RL/selector 训练`

也就是说，数据处理链本身就是算法的一部分。
