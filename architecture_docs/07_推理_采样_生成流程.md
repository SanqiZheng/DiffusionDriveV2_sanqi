# 07 推理、采样、生成流程

## 1. 先说结论

这个项目的“推理”其实有三种语义：

1. 生成器内部的 diffusion 采样
2. selector 的轨迹筛选
3. 离线评测时再用 PDM 对选中结果打分

如果不把这三层拆开，很容易把代码看乱。

## 2. 生成器采样流程

核心文件：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py`
- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_sel.py`

共同核心函数：

- `TrajectoryHead.forward_test_rl`

## 2.1 起点不是纯噪声，而是 anchor

起始状态：

```python
plan_anchor = self.plan_anchor.unsqueeze(0).unsqueeze(0).repeat(...)
diffusion_output = self.norm_odo(plan_anchor)
noise = torch.randn(...)
diffusion_output = self.diffusion_scheduler.add_noise(...)
```

所以生成起点是：

- `anchor + truncated noise`

不是：

- `N(0, I)` 纯白噪声

## 2.2 采样步数

当前代码中：

- 生成器推理常用 `step_num = 2`

也就是说：

- 实际只做两步去噪

这就是论文里“高效 truncated diffusion”落地到代码后的表现。

## 2.3 每一步做什么

每个 diffusion step 的流程：

1. 取当前 noisy trajectory `diffusion_output`
2. 反归一化得到 `noisy_traj_points`
3. 用位置编码 + `plan_anchor_encoder` 得到轨迹 query feature
4. 用 `time_mlp` 编码 timestep
5. 送入 `diff_decoder`
6. 输出 `poses_reg`
7. 取 `poses_reg[..., :2]` 作为 `x_start`
8. 用 `DDIMScheduler_with_logprob.step(...)` 更新到上一时刻

从生成角度看，本质上就是：

- 当前轨迹候选
- 看地图 / agent / ego 条件
- 预测更干净的一版轨迹
- 再由 scheduler 完成一步采样更新

## 2.4 `DDIMScheduler_with_logprob` 的作用

它继承自 `diffusers.schedulers.DDIMScheduler`，但做了两件额外的事：

1. 支持乘性 exploration noise
2. 返回 `log_prob`

返回值：

- `prev_sample`
- `log_prob`
- `prev_sample_mean`

在训练时：

- `log_prob` 用于 RL

在推理时：

- 主要用 `prev_sample` / `prev_sample_mean`

## 3. RL 版推理在做什么

函数：

- `diffusiondrivev2_model_rl.py::TrajectoryHead.forward_test_rl`

输出不是最终部署轨迹，而更像验证代理指标：

- 先生成一批轨迹
- 计算每条轨迹的 PDM reward
- 取最大 reward 做日志
- 再算一个和 GT 的 `trajectory_loss`

所以 RL 版推理更像：

- “训练过程中的验证”

而不是：

- 最终部署 planner

## 4. selector 版推理在做什么

函数：

- `diffusiondrivev2_model_sel.py::TrajectoryHead.forward_test_rl`

这才是更接近最终部署逻辑的版本。

## 4.1 第一步：生成大量候选

代码里：

- `num_groups = 10`
- 每组有 20 个 anchor

所以初始候选数：

- `10 * 20 = 200`

之后又调用：

- `add_mul_noise(diffusion_output)`

默认 `n_aug=3`，函数会返回：

- 原轨迹
- 再加 3 组 multiplicative augmentation

因此推理时最终 coarse scorer 看到的候选数会进一步变多。

## 4.2 第二步：coarse scorer

coarse scorer 会对所有候选预测 5 个子指标头：

- NC
- EP
- DAC
- TTC
- C

再组合成：

- `final_coarse_reward`

并取：

- `best_coarse_flat = argmax(final_coarse_reward)`

得到 coarse best trajectory。

## 4.3 第三步：top-k 细排

然后：

- `topk = 32`

把 coarse top-32 候选送进 fine scorer。

fine scorer 有 3 层，所以会得到：

- 3 个 fine stage best trajectory

最终代码会收集：

- 1 条 coarse best
- 3 条 fine best

组成：

- `traj_to_score.shape = (B, 4, 8, 3)`

## 4.4 第四步：离线评测时再打 PDM 分

接下来代码会：

```python
reward_group, metric_cache, sub_rewards_group, _ = self.get_pdm_score_para(traj_to_score, metric_cache)
```

这一步不是 selector 的选择逻辑本身，而是：

- 为了离线验证，把已选出来的轨迹再喂给 PDM 打真实分数

输出的是：

- `reward_dict`

其中包含：

- `coarse_reward`
- `fine_reward_0`
- `fine_reward_1`
- `fine_reward_2`
- 各种 coarse/fine 子指标均值

## 5. 为什么 README 说官方评测要手动改一行代码

因为当前 selector `forward_test_rl` 默认最后返回的是：

- `{'reward_dict': reward_dict}`

而官方 planner 评测入口 `run_pdm_score.py` 走的是：

- `AbstractAgent.compute_trajectory(...)`

这个接口要求：

- `forward` 返回 `predictions["trajectory"]`

代码里已经留了注释：

```python
# return {"trajectory": traj_to_score[:,-1]}
```

也就是说，作者自己知道：

- 官方评测需要返回最终轨迹
- 当前默认代码为了快速离线评测，保留的是 reward logging 版本

## 6. 快速评测 `run_pdm_score_fast.py` 在做什么

这个脚本不是走 planner API，而是：

1. 直接读 cache 特征和 metric cache
2. 通过 Lightning `validate(...)`
3. 在 `forward + compute_loss` 里把 reward 指标算出来

所以它快的原因是：

- 不需要完整在线模拟输入管线
- 直接复用缓存
- 甚至目标和 metric cache 都已经就位

但这也意味着：

- 它是离线评测捷径
- 不是实际部署时可直接调用的 planner 接口

## 7. sampler / solver / scheduler 在本项目里的角色

## 7.1 diffusion scheduler

真正的 diffusion 采样器是：

- `DDIMScheduler`
- `DDIMScheduler_with_logprob`

它们扮演的就是：

- generative sampler / solver

### `DDIMScheduler`

作用：

- 普通 2-step truncated denoising

### `DDIMScheduler_with_logprob`

作用：

- 同时返回采样后的状态和该步 log probability
- 用于 RL

## 7.2 PDM simulator

文件：

- `navsim/planning/simulation/planner/pdm_planner/simulation/pdm_simulator.py`

作用：

- 把离散轨迹 rollout 成闭环车辆状态序列

## 7.3 PDM scorer

文件：

- `navsim/planning/simulation/planner/pdm_planner/scoring/pdm_scorer.py`

作用：

- 计算 NC / DAC / TTC / Comfort / Progress 等指标

## 8. 从输入或噪声生成最终输出的两条主线

### 8.1 生成器主线

```text
scene feature
  -> anchor + truncated noise
  -> 2-step diffusion denoise
  -> 候选轨迹集合
```

### 8.2 selector 主线

```text
候选轨迹集合
  -> coarse scorer
  -> top-k
  -> fine scorer
  -> 最终轨迹
```

## 9. 条件信息是在推理过程中怎么用的

## 9.1 地图 / 场景条件

通过：

- `cross_bev_attention`

轨迹点直接从 BEV 上采样上下文。

## 9.2 动态目标条件

通过：

- `cross_agent_attention`

候选轨迹会去看周围目标 query。

## 9.3 ego 全局条件

通过：

- `cross_ego_attention`

每条候选轨迹都能读到 ego query 的全局信息。

## 9.4 diffusion 时间条件

通过：

- `time_mlp`
- `ModulationLayer`

每个 denoising step 都有自己的 timestep conditioning。

## 10. 代码里真正被选择出来的是哪条轨迹

严格按当前默认代码：

- 快速评测并没有返回轨迹给上层
- 只是把 coarse best 和 3 个 fine best 的真实 reward 记下来

如果按官方评测所需的改法：

- 返回 `traj_to_score[:, -1]`

也就是：

- 最后一层 fine scorer 选出来的 best trajectory

## 11. 和论文叙述相比，代码推理链有哪些差异

### 11.1 论文说验证时 `eta=0` 做 deterministic inference

代码确实在 DDIM step 时用 `eta=0.0`。  
但是 selector 推理后又调用了：

- `add_mul_noise(diffusion_output)`

这意味着最终给 scorer 的候选集仍然带有随机性增强。

### 11.2 论文语义上 selector 最终输出轨迹

代码默认输出 `reward_dict`，是为了离线评测方便。

## 12. 一句话总结推理流程

DiffusionDriveV2 的推理不是“扩散一步到位”，而是：

先用 anchor-conditioned truncated diffusion 生成一大批候选，再用 coarse-to-fine selector 逐级淘汰，最终选出一条轨迹；离线评测时还会额外调用 PDM 给选出的结果做真实闭环打分。
