# 09 逐文件精读：训练入口与主链路

这一篇的目标很明确：

带你按“真正跑训练时的执行顺序”读源码，而不是按目录乱翻。

---

## 1. 第一个一定要读的文件：`run_training.py`

文件：

- `navsim/planning/script/run_training.py`

这是整个训练系统的总入口。

### 1.1 `build_datasets(cfg, agent)`

职责：

- 根据 Hydra 配置构造训练集和验证集

关键步骤：

1. `instantiate(cfg.train_test_split.scene_filter)` 生成 train / val 的 `SceneFilter`
2. 用 `cfg.train_logs` 和 `cfg.val_logs` 裁剪 log 名单
3. 用 `SceneLoader` 构造场景读取器
4. 用 `Dataset(...)` 包装成 feature/target 数据集

你读这段代码时要带着的问题：

- 训练集和验证集到底是怎么切的
- `SceneLoader` 什么时候读原始日志
- feature builder / target builder 是谁提供的

答案是：

- agent 决定 feature builder 和 target builder
- `run_training.py` 只负责组装，不关心具体模型细节

### 1.2 `main(cfg)`

职责：

- 实例化 agent
- 包装 LightningModule
- 构造 DataLoader
- 启动 `trainer.fit`

执行顺序：

```text
seed_everything
  -> instantiate(cfg.agent)
  -> AgentLightningModule(agent)
  -> Dataset / CacheOnlyDataset
  -> DataLoader
  -> pl.Trainer(...)
  -> trainer.fit(...)
```

如果你只想先抓住主脉络，这个文件就是最好起点。

## 2. 第二个必读文件：`agent_lightning_module.py`

文件：

- `navsim/planning/training/agent_lightning_module.py`

这个文件很短，但非常重要，因为它定义了 batch 如何真正喂给 agent。

### 2.1 `_step(batch, logging_prefix)`

逻辑分叉点：

```python
if len(batch) == 2:
    features, targets = batch
    prediction = self.agent.forward(features, targets)
else:
    features, targets, pdm_token_path, token = batch
    prediction = self.agent.forward(features, targets, pdm_token_path, token)
```

这说明：

- 普通 agent 只吃 `features, targets`
- DiffusionDriveV2 这类 RL / selector agent 还要额外吃 `metric_cache_path` 和 `token`

然后：

```python
loss_dict = self.agent.compute_loss(features, targets, prediction)
return loss_dict['loss']
```

也就是说：

- Lightning 层并不知道 loss 怎么算
- 它只要求 agent 返回一个 `loss_dict`

## 3. 第三个必读文件：`default_training.yaml`

文件：

- `navsim/planning/script/config/training/default_training.yaml`

读这个文件的目的不是看超参数，而是理解 Hydra 装配逻辑。

### 3.1 你最该关注的字段

- `defaults`
- `agent`
- `dataloader.params`
- `trainer.params`
- `cache_path`
- `use_cache_without_dataset`
- `force_cache_computation`

### 3.2 为什么它重要

因为后面命令行都是在改这个文件里的字段，例如：

```bash
python navsim/planning/script/run_training.py \
    agent=diffusiondrivev2_rl_agent \
    ...
```

这意味着：

- 命令行不是“传一个脚本参数”
- 而是在覆写 Hydra 配置树

## 5. 第五个必读文件：RL agent 封装

文件：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_rl_agent.py`

### 5.1 `__init__`

这段最重要的不是构造模型，而是冻结策略：

```python
for name, param in self._transfuser_model.named_parameters():
    if not name.startswith("_trajectory_head"):
        param.requires_grad = False
```

然后：

```python
for name, module in self._transfuser_model.named_modules():
    if name and not name.startswith("_trajectory_head"):
        module.eval()
```

说明：

- RL 阶段只训练 trajectory head
- backbone 与其他头全部冻结

### 5.2 `forward`

```python
return self._transfuser_model(features, targets=targets, eta=1.0, metric_cache=metric_cache, token=token)
```

这里有两个重要信息：

1. RL 训练默认 `eta=1.0`
2. `metric_cache` 被直接传进模型内部，用于 reward 计算

### 5.3 `compute_loss`

RL agent 的 loss 根本不走 `transfuser_loss`

而是：

- 直接从 `predictions['loss']`
- 以及 `reward / sub_rewards`

拼成日志字典。

## 6. 第六个必读文件：selector agent 封装

文件：

- `navsim/agents/diffusiondrivev2/diffusiondrivev2_sel_agent.py`

### 6.1 `__init__`

与 RL agent 不同，这里只打开 scorer 相关参数：

- `plan_anchor_scorer_encoder`
- `scorer_decoder`
- 各种 coarse/fine head

### 6.2 `compute_loss`

selector 的 `predictions` 结构是：

- `loss`
- `sub_loss_dict`
- `reward_dict`

它会把它们并成一个统一的 `loss_dict` 给 Lightning。

这说明：

- selector 阶段的日志会比 RL 阶段更丰富
- 你可以直接看到 coarse/fine 各头的 loss 和 reward

## 7. 第七个必读文件：`diffusiondrivev2_model_rl.py`

这是 Stage I 真正的核心文件。

建议你按下面顺序读：

### 7.1 `V2TransfuserModel.__init__`

看它如何组装：

- backbone
- query embeddings
- tf decoder
- agent head
- trajectory head

### 7.2 `V2TransfuserModel.forward`

看顶层张量流：

- BEV 特征如何变 memory
- query 如何 split 成 ego / agent
- 为什么 `old_pred` 要先无梯度跑一遍

### 7.3 `TrajectoryHead.forward_train_rl`

看：

- anchor 复制
- truncated noise
- rollout
- reward / advantage

### 7.4 `TrajectoryHead.get_rlloss`

看：

- log_prob 如何重算
- RL loss 如何构造
- imitation regularization 如何混合

### 7.5 `TrajectoryHead.forward_test_rl`

看验证时到底记录了什么指标。

---

## 8. 第八个必读文件：`diffusiondrivev2_model_sel.py`

这是 Stage II 真正的核心文件。

建议按下面顺序读：

### 8.1 `TrajectoryHead.__init__`

重点看新增了哪些模块：

- `plan_anchor_scorer_encoder`
- `scorer_decoder`
- `fine_scorer_decoder`
- 各个 coarse/fine 子头

### 8.2 `TrajectoryHead.forward_train_rl`

重点看：

- 为什么一开始 `with torch.no_grad()`
- GTRS vocab 是怎么拼进来的
- coarse/fine 两阶段怎么串起来

### 8.3 `_score_coarse`

看 coarse BCE + rank loss 怎么实现。

### 8.4 `_select_topk`

看 top-k 候选是怎么 gather 出来的。

### 8.5 `_score_fine_multi`

看三层 fine scorer 怎么各自产生 best trajectory。

### 8.6 `forward_test_rl`

看默认推理为什么返回 `reward_dict` 而不是 `trajectory`。

## 9. 第九个必读文件：`run_pdm_score_fast.py`

文件：

- `navsim/planning/script/run_pdm_score_fast.py`

为什么建议读它：

- 它是你理解“离线评测链”最短的入口
- 很容易看清 feature cache、metric cache、Lightning validate 是怎么串起来的

它的本质是：

- 不用 planner API
- 直接把 cache 喂进 agent
- 再让模型内部算 reward

---

## 10. 推荐的实际源码精读顺序

如果你现在就要开始边看边读，建议顺序如下：

1. `run_training.py`
2. `agent_lightning_module.py`
3. `default_training.yaml`
4. `diffusiondrivev2_rl_agent.py`
5. `diffusiondrivev2_model_rl.py`
6. `diffusiondrivev2_sel_agent.py`
7. `diffusiondrivev2_model_sel.py`
8. `run_pdm_score_fast.py`

为什么这样排：

- 先看外层壳子，知道调用关系
- 再看 RL 主链
- 最后看 selector 主链

## 11. 这一条主链里最值得做的笔记

建议你自己边读边记下这 6 个问题：

1. `features / targets / metric_cache / token` 分别从哪里来
2. `_trajectory_head` 为什么要先无梯度 rollout 再回放
3. `advantages` 的 shape 为什么是 `(B, num_groups*20, step_num)`
4. selector 的监督信号究竟来自真实 GT 还是 PDM reward
5. 为什么快速评测能比官方评测快很多
6. 为什么官方评测要手动改 `return {"trajectory": ...}` 那行

如果这 6 个问题都能自己答出来，这条训练主链你就已经吃透一大半了。
