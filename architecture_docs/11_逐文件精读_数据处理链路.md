# 11 逐文件精读：数据处理链路

这一篇只做一件事：

把数据链从原始日志一路追到模型输入与 reward cache。

## 1. `navsim/common/dataclasses.py`

这是整个数据系统的底座。

## 1.1 `Camera` / `Cameras`

作用：

- 封装单个相机和多相机输入

你最该看的是：

- `Cameras.from_camera_dict(...)`

因为它决定了：

- 相机图是怎么从日志路径读出来的
- 内参外参是否随样本一起进入内存

## 1.2 `Lidar`

作用：

- 封装 merged LiDAR point cloud

关键函数：

- `Lidar.from_paths(...)`

结论：

- 模型真正看到的 LiDAR 不是原始 range image
- 而是后面 feature builder 再投成二维 histogram

## 1.3 `EgoStatus`

它保存：

- `ego_pose`
- `ego_velocity`
- `ego_acceleration`
- `driving_command`

你要注意：

- `AgentInput` 里的 ego pose 是相对坐标
- `Scene.frames` 里的 ego status 则可以在全局语义下理解

## 1.4 `AgentInput.from_scene_dict_list`

这是“输入侧样本”构造的关键函数。

它会：

1. 先把历史帧的全局 ego pose 取出来
2. 转成以当前帧为原点的局部坐标
3. 依次装入：
   - `ego_statuses`
   - `cameras`
   - `lidars`

所以模型输入从一开始就是：

- 以当前 ego 为参考系的局部表示

## 1.5 `Scene.get_future_trajectory`

这是 target trajectory 的来源。

它会：

1. 取历史最后一帧作为原点
2. 把未来 ego 轨迹转成相对坐标

所以监督目标 `trajectory` 的语义是：

- 未来轨迹，相对于当前 ego rear axle 的局部坐标

---

## 2. `navsim/common/dataloader.py`

这个文件负责：

- 按日志切样本
- 把 token 管理起来

## 2.1 `filter_scenes`

这是样本切分核心。

工作方式：

1. 读取每个 log pickle
2. 用滑窗 `split_list(...)` 切出固定长度帧序列
3. 根据 route / token / max_scenes 做过滤
4. 用历史最后一帧 token 作为样本键

你要特别记住：

- 一个 token 对应的不是单帧，而是一整个场景切片

## 2.2 `SceneLoader`

关键方法：

- `get_scene_from_token(token)`
- `get_agent_input_from_token(token)`
- `get_tokens_list_per_log()`

如果你想手动 debug 某个样本，`SceneLoader` 是最好入口。

## 2.3 `MetricCacheLoader`

作用：

- 读取 `metric_cache.pkl`

为什么它关键：

- RL reward 和快速评测都依赖它

它说明这个项目不是纯监督学习数据流，而是：

- feature/target cache
- metric cache

两条并行数据流同时存在。

---

## 3. `transfuser_features.py`

文件：

- `navsim/agents/diffusiondrivev2/transfuser_features.py`

## 3.1 `TransfuserFeatureBuilder.compute_features`

输出 3 个字段：

- `camera_feature`
- `lidar_feature`
- `status_feature`

这也决定了模型输入 dict 的键名。

## 3.2 `_get_camera_feature`

这里最值得注意的是：

- 不是全 8 路相机都喂进去
- 只取 `l0 / f0 / r0`
- 再拼成一张超宽图

这是一种很工程化的“便宜视野扩展”。

## 3.3 `_get_lidar_feature`

LiDAR 处理也很工程化：

- 不做点级 transformer
- 不做 voxel encoder
- 直接 rasterize 成 BEV histogram

好处：

- 简单
- 稳定
- 和 TransFuser 风格一致

## 3.4 `TransfuserTargetBuilder.compute_targets`

输出：

- `trajectory`
- `agent_states`
- `agent_labels`
- `bev_semantic_map`

这意味着一个 builder 同时负责：

- 规划监督
- 目标框监督
- 语义地图监督

虽然在 V2 训练里，真正驱动主创新的是 trajectory 与 reward，但这些 target 依然被保留下来。

---

## 4. `navsim/planning/training/dataset.py`

这个文件是整个数据链最值得精读的文件之一。

## 4.1 `Dataset`

职责：

- 从 `SceneLoader` 在线计算 feature/target
- 或读取已有 cache

最重要的方法：

- `_cache_scene_with_token`
- `_load_scene_with_token`
- `cache_dataset`
- `__getitem__`

### 你需要弄懂的点

1. cache 的目录结构是什么
2. 什么时候会在线算 feature
3. 什么时候完全走 cache

## 4.2 `CacheOnlyDataset`

这是 DiffusionDriveV2 训练最关键的数据集包装器。

因为它不只返回：

- `features`
- `targets`

还返回：

- `pdm_token_path`
- `token`

这两个字段分别干嘛：

- `pdm_token_path`
  - 让模型在训练时能去 PDM scorer 取 reward
- `token`
  - selector 训练时查 GTRS 词表分数

## 4.3 `CacheOnlyDatasetTest`

这是快速评测脚本专用版本。

区别：

- 训练时 `pdm_token_path` 是从训练 cache 路径推出来的
- 测试时 `metric_cache_path` 是单独显式提供的

---

## 5. `run_dataset_caching.py`

这个脚本的意义不是训练，而是：

- 先把原始场景全部跑一遍
- 把 feature/target 预存下来

为什么重要：

- 否则训练过程中在线做相机读取、LiDAR 投影、地图 rasterization 会非常慢

---

## 6. `run_pdm_score_fast.py`

虽然它是评测脚本，但也属于数据链的一部分，因为它定义了：

- feature cache
- metric cache

是如何在验证时重新对齐的。

这里最关键的函数是：

- `custom_collate_fn`

它把 batch 组织成：

- `features`
- `targets`
- `metrics`
- `token`

和训练阶段 `CacheOnlyDataset` 的 4 元组语义保持一致。

---

## 7. 真正建议你手动打印的几个对象

如果你要自己开个 notebook 或 debug 脚本，最建议打印的是：

1. `AgentInput.ego_statuses[-1]`
2. `features["camera_feature"].shape`
3. `features["lidar_feature"].shape`
4. `targets["trajectory"]`
5. `targets["agent_states"].shape`
6. `targets["bev_semantic_map"].shape`
7. `pdm_token_path`

把这几个对象看明白后，后面模型里的大部分 shape 都不会迷糊。

---

## 8. 数据链里最容易忽略但很关键的两点

### 8.1 target trajectory 只有 8 个点

虽然 `SceneFilter.num_future_frames = 10`，但 target builder 实际只按：

- `trajectory_sampling.num_poses = 8`

取未来轨迹。

说明：

- 样本里未来帧比监督轨迹更多
- 但模型预测目标只用前 8 个 future poses

### 8.2 reward 依赖另一条“隐藏数据流”

普通监督学习里，batch 通常只需要：

- input
- target

这里不是。

这里还需要：

- `metric_cache_path`

所以如果你以后自己迁移这个项目，最容易漏掉的不是 feature builder，而是：

- reward cache 这条并行链

---

## 9. 数据处理链路的一句话总结

这个工程的数据系统本质上分成两部分：

1. `Scene -> feature/target`，服务于感知与轨迹监督
2. `token -> metric_cache`，服务于闭环 reward 与离线评测

而 DiffusionDriveV2 的创新恰恰建立在这两条数据流同时存在之上。
