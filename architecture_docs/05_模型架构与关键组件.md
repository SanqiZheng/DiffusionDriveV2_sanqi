# 05 模型架构与关键组件

## 1. 总体结构

从代码实现看，DiffusionDriveV2 的主模型可以分成 5 个模块：

1. `TransfuserBackbone`
2. BEV memory / query decoder
3. `AgentHead`
4. `TrajectoryHead` 里的扩散生成器
5. `TrajectoryHead` 里的 coarse-to-fine selector

对应文件：

- `navsim/agents/diffusiondrivev2/transfuser_backbone.py`
- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py`
- `navsim/agents/diffusiondrivev2/diffusiondrivev2_model_sel.py`
- `navsim/agents/diffusiondrivev2/modules/blocks.py`

## 2. 顶层 `V2TransfuserModel`

文件：

- RL 版：`navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py`
- selector 版：`navsim/agents/diffusiondrivev2/diffusiondrivev2_model_sel.py`

核心成员：

- `_backbone`
- `_bev_downscale`
- `_status_encoding`
- `_keyval_embedding`
- `_query_embedding`
- `_tf_decoder`
- `_agent_head`
- `_trajectory_head`
- `bev_proj`

## 2.1 输入输出接口

输入：

- `camera_feature`
- `lidar_feature`
- `status_feature`

输出：

- `bev_semantic_map`
- `agent_states`
- `agent_labels`
- RL 版训练时：`loss`, `reward`, `sub_rewards`
- selector 版训练时：`loss`, `sub_loss_dict`, `reward_dict`

## 2.2 顶层张量流

默认 shape 可以按下面理解：

| 张量 | shape | 来源 |
| --- | --- | --- |
| `camera_feature` | `(B, 3, 256, 1024)` | 三路相机拼接 |
| `lidar_feature` | `(B, 1, 256, 256)` | LiDAR 直方图 |
| `status_feature` | `(B, 8)` | ego 状态 |
| `bev_feature_upscale` | `(B, 64, 64, 64)` | backbone top-down 输出 |
| `bev_feature` | `(B, 512, 8, 8)` | backbone fused BEV 特征 |
| `keyval` | `(B, 65, 256)` | 64 个 BEV token + 1 个 status token |
| `query` | `(B, 31, 256)` | 1 个 ego query + 30 个 agent query |
| `trajectory_query` | `(B, 1, 256)` | ego trajectory query |
| `agents_query` | `(B, 30, 256)` | agent query |

这个形状为什么成立：

- `bev_feature` 被 `_bev_downscale: Conv2d(512 -> 256)` 压到 256 维
- `8 x 8 = 64` 个 BEV token
- 再加 1 个 `status token`

所以：

- `keyval_embedding = nn.Embedding(8**2 + 1, 256)`

## 3. `TransfuserBackbone`

文件：

- `navsim/agents/diffusiondrivev2/transfuser_backbone.py`

作用：

- 编码图像与 LiDAR
- 在多尺度上做跨模态融合
- 产出供后续查询解码器使用的 BEV 特征

### 3.1 图像编码器

成员：

- `self.image_encoder = timm.create_model(config.image_architecture, pretrained=True, features_only=True)`

默认：

- `image_architecture = "resnet34"`

### 3.2 LiDAR 编码器

成员：

- `self.lidar_encoder = timm.create_model(..., in_chans=in_channels, pretrained=False, features_only=True)`

默认：

- `in_channels = 1`

### 3.3 跨模态融合

关键模块：

- `self.transformers`
- `self.lidar_channel_to_img`
- `self.img_channel_to_lidar`

工作方式：

1. 图像 feature 和 LiDAR feature 先各自通过 backbone block
2. 在每个尺度上用 `GPT` 模块融合
3. 融合后的结果再回写到两个分支

直觉上，这不是简单 concat，而是“在每一层都让两个模态交换上下文”。

### 3.4 top-down FPN

关键函数：

- `TransfuserBackbone.top_down`

作用：

- 把最后一层 LiDAR BEV 特征上采样成更高分辨率的 BEV 特征图

默认输出：

- `features` 约为 `(B, 64, 64, 64)`

这个高分辨率 feature 后面会作为：

- `cross_bev_feature`

给轨迹生成器和 selector 做轨迹-地图交互。

## 4. query decoder 与条件记忆

顶层 `V2TransfuserModel.forward` 做了两件很关键的事情：

### 4.1 把 BEV 特征变成 token memory

过程：

1. `bev_feature = self._bev_downscale(bev_feature)` 变成 `(B, 256, 8, 8)`
2. `flatten(-2, -1).permute(0, 2, 1)` 变成 `(B, 64, 256)`
3. `status_feature` 过 `self._status_encoding` 变成 `(B, 256)`
4. 与 BEV token 拼接成 `(B, 65, 256)`

这相当于：

- 把整张 BEV 图压成 64 个 token
- 再加一个专门的 ego 状态 token

### 4.2 用 learnable query 解出 ego 和 agent 槽位

过程：

1. `query = self._query_embedding.weight[None, ...].repeat(batch_size, 1, 1)`
2. 送入 `_tf_decoder(query, keyval)`
3. 输出 `query_out`
4. 再按 `[1, 30]` 切成：
   - `trajectory_query`
   - `agents_query`

这一步非常像 DETR 思路：

- memory 是场景上下文
- query 是任务槽位

## 5. `AgentHead`

文件：

- RL 版与 selector 版都在各自模型文件里定义了 `AgentHead`

作用：

- 预测周围车辆框

输入：

- `agents_query`，shape `(B, 30, 256)`

输出：

- `agent_states`，shape `(B, 30, 5)`
- `agent_labels`，shape `(B, 30)`

后处理：

- `(x, y)` 用 `tanh() * 32`
- `heading` 用 `tanh() * pi`

说明：

- agent detection 不是这篇论文主创新，但它是完整 planner 的辅助输出

## 6. `TrajectoryHead`：真正的核心

这部分才是 DiffusionDriveV2 的灵魂。

在 RL 版和 selector 版里都有 `TrajectoryHead`，但职责不同：

- RL 版：负责生成与 RL 训练
- selector 版：负责生成候选并训练 selector

共同组件包括：

- `plan_anchor`
- `plan_anchor_encoder`
- `time_mlp`
- `diffusion_scheduler`
- `diffusionrl_scheduler`
- `diff_decoder`

## 6.1 `plan_anchor`

`plan_anchor` 是整个多模态结构的根。

形状：

- `(20, 8, 2)`

含义：

- 20 个意图 anchor
- 每个 anchor 是未来 8 个轨迹点的 `(x, y)`

## 6.2 `norm_odo` / `denorm_odo`

文件：

- RL 版与 selector 版 `TrajectoryHead`

作用：

- 在 diffusion 空间和真实轨迹坐标空间之间变换

当前实现：

- `x` 缩放到 `x / 50`
- `y` 缩放到 `y / 20`
- heading 也留了接口，但主生成过程实际只处理 `xy`

这说明：

- diffusion 实际在“归一化轨迹坐标空间”里工作

## 6.3 `time_mlp`

作用：

- 把 diffusion timestep 编码成条件向量

结构：

- `SinusoidalPosEmb`
- `Linear`
- `Mish`
- `Linear`

输出：

- `(B, 256)` 或 reshape 后 `(B, 1, 256)`

这是标准 diffusion 里的 timestep embedding。

## 6.4 `plan_anchor_encoder`

作用：

- 把当前 noisy trajectory 的位置编码投到模型隐空间

过程：

1. `gen_sineembed_for_position(noisy_traj_points, hidden_dim=64)`
2. `flatten(-2)`，把 8 个点的编码拼起来
3. 过 MLP

直觉上：

- 当前轨迹形状本身被编码成 query feature

## 7. 轨迹生成解码器 `CustomTransformerDecoder`

关键类：

- `CustomTransformerDecoderLayer`
- `CustomTransformerDecoder`

## 7.1 每层做了什么

一层 decoder 大致流程：

1. `cross_bev_attention`
   - 轨迹点去 BEV 特征图上采样
2. `cross_agent_attention`
   - 看周围 agent
3. `cross_ego_attention`
   - 看 ego query
4. `ffn`
5. `time_modulation`
   - 用 diffusion timestep 调制
6. `task_decoder`
   - 输出轨迹偏移和 mode logit

### 条件注入位置

这是看源码时最值得盯住的一点：

#### 地图 / BEV 条件

注入点：

- `GridSampleCrossBEVAttention`

方式：

- 用当前轨迹点坐标去 BEV feature 图上 `grid_sample`

这不是全局 attention，而是“沿轨迹采样地图上下文”。

#### agent 条件

注入点：

- `cross_agent_attention`

方式：

- 轨迹 query attend 到 `agents_query`

#### ego 全局条件

注入点：

- `cross_ego_attention`

方式：

- 轨迹 query attend 到 `ego_query`

#### timestep 条件

注入点：

- `ModulationLayer.time_modulation`

方式：

- 生成 scale / shift，对轨迹 feature 做 FiLM 风格调制

## 7.2 `GridSampleCrossBEVAttention`

文件：

- `navsim/agents/diffusiondrivev2/modules/blocks.py`

作用：

- 把轨迹点位置变成对 BEV feature 的条件读取

输入：

- `queries`: `(B, G, C)`
- `traj_points`: `(B, G, 8, 2)`
- `bev_feature`: `(B, C_bev, H, W)`

过程：

1. 轨迹坐标归一化到 `[-1, 1]`
2. 用 `attention_weights` 给 8 个轨迹点分配权重
3. 用 `grid_sample` 在 BEV 上采样
4. 加权求和
5. 线性投影回 query space

直觉：

- 当前候选轨迹会主动去“看一眼”它经过的位置在地图上长什么样

## 8. RL 版 `TrajectoryHead`

关键函数：

- `forward_train_rl`
- `get_rlloss`
- `forward_test_rl`

### 8.1 `forward_train_rl`

职责：

- 用当前模型 rollout 一批轨迹
- 计算 reward
- 构造 advantage

输出：

- `all_diffusion_output`
- `advantages`
- `reward`
- `sub_rewards`

### 8.2 `get_rlloss`

职责：

- 对刚才 rollout 得到的 diffusion chain 重新计算 log_prob
- 形成 policy gradient loss
- 再加 imitation regularization

这就是代码里“先无梯度采样，再有梯度回放”的核心套路。

### 8.3 `forward_test_rl`

职责：

- 做 2-step 轨迹采样
- 在验证时用 PDM reward 做日志

注意：

- RL agent 不是最终部署 agent
- 它更像 Stage I 训练产物

## 9. selector 版 `TrajectoryHead`

额外组件：

- `plan_anchor_scorer_encoder`
- `scorer_decoder`
- `fine_scorer_decoder`
- 5 个 coarse 头
- 5 个 fine 头

## 9.1 coarse scorer

作用：

- 对大量候选先做一次粗筛

输出子头：

- `NC_head`
- `EP_head`
- `DAC_head`
- `TTC_head`
- `C_head`

组合总分：

```text
sigmoid(NC) * sigmoid(DAC) *
(5 * sigmoid(TTC) + 5 * sigmoid(EP) + 2 * sigmoid(C)) / 12
```

这与 PDM 指标的权重结构是一致的。

## 9.2 fine scorer

作用：

- 对 coarse top-k 继续细排

结构：

- `fine_scorer_decoder` 有 3 层
- 每层都能输出一个 best trajectory

这解释了为什么推理时会出现：

- 1 个 coarse best
- 3 个 fine best

## 10. selector 的 GTRS 增强

selector 版还有一块论文里提到、代码也实现了的增强：

- `get_vocab_pdm_subscores`

相关文件：

- `gtrs_traj/16384.npy`
- `gtrs_traj/navtrain_16384.pkl`

作用：

- 给 selector 训练补充固定轨迹词表及其 PDM 子指标

直觉：

- 不只让 selector 看模型自己生成的轨迹
- 还让它接触更大词表里的候选，提高打分鲁棒性

## 11. 哪些组件是“真正在用”的，哪些更像遗留接口

真正在主链上起作用的：

- `TransfuserBackbone`
- `GridSampleCrossBEVAttention`
- `TrajectoryHead`
- `DDIMScheduler`
- `DDIMScheduler_with_logprob`
- coarse/fine scorer

更像遗留或未真正接入主链的：

- `ConditionalUnet1D`
- `LossComputer` 在 V2 中被实例化但核心训练没直接用
- `transfuser_loss.py` 在 V2 agent 中没有走主损失路径

## 12. 一句话总结模型结构

这个项目的模型不是“一个 U-Net”，而是：

`TransFuser 风格多模态感知 backbone + DETR 风格 query 解码 + anchor-conditioned truncated diffusion trajectory decoder + coarse-to-fine trajectory scorer`

真正决定论文创新是否落地的，不在 backbone，而在 `TrajectoryHead`。
