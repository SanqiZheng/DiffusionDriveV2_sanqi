# 10 逐文件精读：核心模型模块

这一篇只盯核心模型文件，不再重复训练入口。

## 1. `transfuser_backbone.py`

文件：

- `navsim/agents/diffusiondrivev2/transfuser_backbone.py`

### 1.1 读这个文件时的目标

你不是要看懂每个卷积，而是要搞清楚三件事：

1. 图像和 LiDAR 是如何融合的
2. 最后给轨迹生成器的 BEV memory 长什么样
3. 为什么 query decoder 输入维度是 `65 x 256`

### 1.2 最关键的函数

- `TransfuserBackbone.__init__`
- `TransfuserBackbone.forward`
- `TransfuserBackbone.fuse_features`

### 1.3 重点观察

- `timm.create_model(... features_only=True)`
- `self.transformers`
- `self.avgpool_img`
- `self.avgpool_lidar`
- `self.top_down`

你要形成的直觉是：

- backbone 输出的不只是一个全局向量
- 而是可供轨迹点去查询的 BEV feature map

---

## 2. `modules/blocks.py`

文件：

- `navsim/agents/diffusiondrivev2/modules/blocks.py`

这个文件虽然短，但非常关键。

### 2.1 `gen_sineembed_for_position`

作用：

- 给轨迹点 `(x, y)` 做正弦位置编码

后面这套编码会被：

- `plan_anchor_encoder`
- `plan_anchor_scorer_encoder`

直接拿来做轨迹特征输入。

### 2.2 `GridSampleCrossBEVAttention`

这是生成器最值得盯的一层。

核心思想：

- 不是让 query 和整张地图全局 attention
- 而是让轨迹点直接去地图 feature 图上采样

你可以把它理解成：

- “如果这条轨迹真的要经过这些坐标点，那这些位置对应的 BEV 地图特征是什么”

### 2.3 `GridSampleCrossBEVAttentionScorer`

和上面结构相似，但给 selector 用。

区别：

- 输出维度更大
- 面向打分器而不是生成器

## 3. `diffusiondrivev2_model_rl.py`

这是 Stage I 的主战场。

## 3.1 顶层 `V2TransfuserModel`

### 先看 `__init__`

你要特别注意这些成员：

- `_keyval_embedding`
- `_query_embedding`
- `_bev_downscale`
- `_status_encoding`
- `_tf_decoder`
- `_trajectory_head`

这套结构说明：

- backbone 产出的不是最终轨迹
- 而是给 query decoder 用的 memory

### 再看 `forward`

这里是顶层总装配：

1. backbone 编码
2. BEV flatten 成 token
3. status 变成额外 token
4. query decoder 解出 `trajectory_query` 和 `agents_query`
5. 把它们送给 `TrajectoryHead`

这段 forward 读懂后，你就会知道：

- 轨迹生成不是直接从 backbone feature 上回归
- 而是要先经过 query-level 表示重整

## 3.2 `TrajectoryHead.__init__`

这部分最重要的对象有：

- `self.plan_anchor`
- `self.diffusion_scheduler`
- `self.diffusionrl_scheduler`
- `self.plan_anchor_encoder`
- `self.time_mlp`
- `self.diff_decoder`
- `self._pdm_pool`

你读这里时要注意两件事：

1. 生成器不是纯神经网络，还带着 PDM 评测外环
2. `TrajectoryHead` 其实同时承担了：
   - sampler
   - RL trainer
   - reward bridge

## 3.3 `CustomTransformerDecoderLayer`

这一层是 diffusion denoiser 的核心。

推荐按下面顺序看：

1. `cross_bev_attention`
2. `cross_agent_attention`
3. `cross_ego_attention`
4. `ffn`
5. `time_modulation`
6. `task_decoder`

### 你要特别抓住的一点

条件信息不是一次性拼接，而是分层注入：

- 地图条件通过 `cross_bev_attention`
- 周围车条件通过 `cross_agent_attention`
- ego 全局条件通过 `cross_ego_attention`
- diffusion step 条件通过 `time_modulation`

这是这个项目最像“条件生成模型”的地方。

## 3.4 `DDIMScheduler_with_logprob`

这段代码不长，但非常值得细读。

为什么值得看：

- 它决定了训练时“策略分布”到底怎么定义
- 也决定了 `log_prob` 是怎么被写出来的

阅读重点：

1. `prediction_type == "sample"`
2. `prev_sample_mean`
3. 乘性噪声的生成
4. `log_prob` 的表达式

读完后你应该问自己：

- 这个 `log_prob` 和采样噪声分布是否完全一致

这个问题会在 `13_实现观察_潜在问题与二次开发建议.md` 里继续展开。

## 3.5 `forward_train_rl`

这是最值得逐行读的函数之一。

建议你自己边看边在纸上记：

- `bs`
- `num_groups`
- `ego_fut_mode`
- `G_all`
- `step_num`

并在每一行后面写 shape。

最重要的 6 个片段：

1. anchor 扩展成样本组
2. truncated noise
3. denoising rollout
4. `bezier_xyyaw`
5. `get_pdm_score_para`
6. advantage 计算

## 3.6 `get_rlloss`

这是第二个必须逐行啃的函数。

它做的事情可以概括成：

- 重新走一遍 rollout 的每个 denoising step
- 把每一步的 log_prob 都算出来
- 再按 advantage 加权

它本质上是：

- 一个 diffusion-policy 的 trajectory-level reinforce 实现

## 4. `diffusiondrivev2_model_sel.py`

这是 Stage II 的主战场。

## 4.1 为什么这个文件比 RL 版更复杂

因为它实际上包含了两套网络：

1. 生成器
2. 选择器

虽然训练时生成器冻结，但代码依然要先跑生成器才能拿到候选。

## 4.2 `TrajectoryHead.__init__`

这一段最该关注的是新增模块：

- `plan_anchor_scorer_encoder`
- `scorer_decoder`
- `fine_scorer_decoder`
- coarse/fine 各种 head
- `rank_loss`

你要形成的认识是：

- selector 不是在 BEV feature 上直接打分
- 而是以“轨迹”为 query，再回头读地图和 agent 上下文

## 4.3 `_get_scorer_inputs`

这个函数很值得单独看，因为它回答了：

- selector 输入到底是什么

它做了三件事：

1. 把轨迹重新归一化 / 反归一化
2. 对 `(x, y)` 做正弦编码
3. 对 heading 做 1D 编码

所以 selector 真正看的不是原始坐标，而是：

- 带位置与朝向编码的轨迹表示

## 4.4 `_score_coarse`

粗排的关键文件点。

建议你边看边回答三个问题：

1. 为什么要把总分拆成多个子指标头
2. 为什么 `progress` 额外加 rank loss
3. 为什么 `comfort` 要做 mask

## 4.5 `_select_topk`

这段代码是 coarse-to-fine 的桥梁。

它的意义不在数学，而在工程：

- 如果不先 coarse 筛掉大部分候选
- fine scorer 的计算成本会很高

## 4.6 `_score_fine_multi`

细排不是一个 head，而是 3 层解码器反复 refine。

所以推理时你看到的不是一个 final scorer，而是：

- 第 1 层 fine 的最优结果
- 第 2 层 fine 的最优结果
- 第 3 层 fine 的最优结果

## 4.7 `forward_test_rl`

这是另一个最值得读的函数。

因为它能让你真正理解：

- 代码默认的“推理”到底返回什么
- 为什么官方评测要手动改一行

## 5. `transfuser_features.py`

这个文件虽然已经在数据流章节讲过，但从模型视角也值得回头看一遍。

为什么：

- 它定义了模型看到世界的方式

你至少要清楚：

- 为什么相机只用 `cam_l0 / cam_f0 / cam_r0`
- 为什么 LiDAR 只做 2D histogram
- 为什么 target trajectory 只取 8 个未来点

文件：

- `navsim/agents/diffusiondrivev2/modules/scheduler.py`

虽然和算法核心关系没那么强，但实际训练依赖它。

`WarmupCosLR` 的作用：

- warmup

## 7. `transfuser_model_v2.py`：为什么也建议对照读

文件：

- `navsim/agents/diffusiondrive/transfuser_model_v2.py`

这是原版 DiffusionDrive 的主模型。

建议你对照读的原因：

- 你能很快看出 V2 究竟改了什么

最关键差异：

1. 原版训练是 imitation-style trajectory loss
2. V2 RL 版训练变成 reward/advantage/logprob
3. V2 selector 版又新增了 coarse/fine scorer

如果你先把原版 `forward_train` 看懂，再回来看 `forward_train_rl`，会清楚很多。

---

## 8. 精读核心模型时的建议问题清单

建议你边看边回答：

1. `trajectory_query` 在后面到底起什么作用
2. 为什么 query 只有 1 个 ego query，但还能生成很多轨迹候选
3. noisy trajectory 是如何重新变成 query feature 的
4. `cross_bev_attention` 为什么比普通 self-attention 更贴合规划问题
5. selector 为什么不直接用生成器内部的 `poses_cls`
6. 最终部署时到底应该返回 coarse best 还是最后一层 fine best

如果这几个问题都能回答，核心模型模块就已经基本读通了。
