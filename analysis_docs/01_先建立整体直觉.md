# 01. 先建立整体直觉

## 1. 这份代码到底在做什么

如果你把整个系统想成“输入传感器，输出未来轨迹”的黑盒，那么这份 RL 版本代码做的事情可以拆成四步：

1. 用 TransFuser backbone 把相机、激光雷达、ego 状态编码成场景特征。
2. 从一组固定的轨迹 anchor 出发，加一点噪声，得到很多“候选起点”。
3. 用一个条件去噪网络逐步把这些带噪轨迹修正成合理的未来轨迹。
4. 用 PDM simulator + PDM scorer 给这些轨迹打分，再把分数回传成 RL loss。

所以它不是“直接回归一条轨迹”，而是在“多种驾驶意图的轨迹空间里采样，再筛优，再反向强化更好的采样策略”。

## 2. 对初学者最重要的扩散模型回忆

你现在只记得“加噪”和“去噪”，其实已经够了。这里把它恢复成项目语言。

### 2.1 标准扩散直觉

标准 diffusion 的训练逻辑是：

1. 先拿一个真实样本 `x0`。
2. 往里加噪，得到 `xt`。
3. 模型看见 `xt` 和时间步 `t`，学会恢复真实样本。

常见有三种预测目标：

- 预测噪声 `epsilon`
- 预测速度 `v`
- 直接预测干净样本 `x_start`

这个项目用的是第三种。

### 2.2 这个项目里的“样本”是什么

这里的样本不是图片，不是语音，而是未来轨迹点序列。

更具体地说，主扩散空间里真正被预测的是：

- 未来 `8` 个时刻的 `(x, y)` 轨迹点
- 先经过归一化
- 最后再通过 `bezier_xyyaw` 推出 yaw

也就是说，模型并不是直接在扩散空间里生成 `(x, y, yaw)`，而是先生成 `xy`，再从曲线导数恢复 yaw。

### 2.3 这个项目为什么不从纯噪声开始

如果从纯高斯噪声开始，轨迹空间太大，规划问题太容易塌到保守解。于是作者继承了 DiffusionDrive 的核心思想：

1. 先用 KMeans 从训练集轨迹里聚出 `20` 个 anchor。
2. 每个 anchor 对应一种驾驶意图，比如直行、变道、转弯。
3. 扩散不从“白噪声”起步，而从“anchor 附近的带噪轨迹”起步。

这相当于先把动作空间粗分区，再在每个区内部做细化搜索。

## 3. 为什么这份代码要引入 RL

README 和论文都在强调一个矛盾：

- 纯 imitation learning 很容易只监督“离 GT 最近的那个 mode”
- 其余 mode 没有被真正约束
- 结果是模型虽然会生成很多不同轨迹，但里面混着不少碰撞、越界、低质量解

RL 想解决的是：

1. 不只奖励“像 GT”，还要奖励“闭环表现好”。
2. 让危险轨迹真的被惩罚，而不是没人管。
3. 让模型可以探索出比监督目标更优的轨迹。

所以这个 RL 版本的核心变化，不是换 backbone，而是把“采样轨迹 -> 用环境指标打分 -> 用策略梯度更新”这条链补上。

## 4. 这份代码里的 3 个关键对象

### 4.1 Anchor

来源在 [diffusiondrivev2_model_rl.py](../navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py) 第 731-738 行附近。

含义：

- `plan_anchor` 的 shape 是 `(20, 8, 2)`
- 20 表示 20 个驾驶意图原型
- 8 表示未来 8 个采样点
- 2 表示 `(x, y)`

物理意义：

- 每条 anchor 都像一个“粗规划模板”
- 模型不是无中生有，而是在模板附近做 refinement

### 4.2 Group

来源在同文件第 845-850 行附近。

`num_groups` 不是“20 个 anchor 的数量”，而是“每个 anchor 复制几次做独立探索”。

默认配置在 [diffusiondrivev2_rl_config.py](../navsim/agents/diffusiondrivev2/diffusiondrivev2_rl_config.py) 第 37 行：

- `num_groups = 4`

所以训练时总候选数是：

`4 groups * 20 anchors = 80 trajectories`

直觉上，这就像“同一个驾驶意图，多掷几次骰子，看哪条附近轨迹更好”。

### 4.3 Reward

reward 不是一个神经网络自己随便学出来的分，而是来自：

- [pdm_simulator.py](../navsim/planning/simulation/planner/pdm_planner/simulation/pdm_simulator.py)
- [pdm_scorer.py](../navsim/planning/simulation/planner/pdm_planner/scoring/pdm_scorer.py)
- [pdm_score.py](../navsim/evaluate/pdm_score.py)

主要看这些指标：

- `no_collision`
- `drivable_area`
- `progress`
- `ttc`
- `comfort`
- `driving_direction`
- `final`

这和“只看 L1 轨迹误差”非常不同，因为它更接近真正驾驶质量。

## 5. 这个 RL 版本最该记住的 4 个设计

### 5.1 预测目标是 `x_start`

这意味着模型不是学“这一步噪声是多少”，而是学“当前 noisy trajectory 对应的干净轨迹应该是什么”。

优点：

- 对轨迹这种低维结构化输出更直观
- 直接对应最终几何结果

### 5.2 条件注入是分层的

在 `CustomTransformerDecoderLayer` 里，条件按下面顺序进入：

1. 先从 noisy trajectory 点到 BEV feature 图上取局部地图特征
2. 再看周围 agent token
3. 再看 ego query
4. 最后用 diffusion timestep 做 modulation

这是非常“规划化”的设计，不是纯黑盒 transformer。

### 5.3 推理时步数很少

训练里 `step_num = 10`，测试里 `step_num = 2`，都是截断扩散，不是上百步那种慢采样。

这说明作者的重点不是追求最标准的 diffusion 形式，而是要让规划器足够快。

### 5.4 yaw 不是直接回归，而是由曲线导数得到

对应 [diffusiondrivev2_model_rl.py](../navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py) 第 1158 行开始的 `bezier_xyyaw`。

这么做的动机很强：

- 如果直接预测 yaw，容易出现角度跳变
- 如果先预测平滑 xy，再从切线方向得到 yaw，更自然

## 6. 先给你一个完整心智图

你可以先把这份代码理解成下面这句话：

“相机和激光雷达先提供场景理解，20 条 anchor 提供驾驶意图，多组随机噪声提供探索，条件去噪网络负责把 noisy trajectory 修成可行轨迹，PDM scorer 负责告诉模型哪些轨迹真正在闭环里更安全、更顺、更有效率，最后 RL loss 让模型更偏向生成这些高质量轨迹。”

只要先记住这张图，后面读每个函数就不会迷路。
