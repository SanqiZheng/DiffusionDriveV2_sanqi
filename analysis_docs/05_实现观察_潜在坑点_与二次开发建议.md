# 05. 实现观察、潜在坑点与二次开发建议

这一篇专门讲“代码真实长什么样”，包括未使用接口、和论文不完全一致的地方、以及你以后改代码时最值得小心的点。

## 1. 未使用或弱使用的接口

### 1.1 `ConditionalUnet1D` 被 import 但主链路没用

位置：

- [diffusiondrivev2_model_rl.py](../navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py) 第 12 行

现状：

- import 了 `ConditionalUnet1D`
- 但当前 RL 主链路没有实例化它

推测：

- 这是从老版 DiffusionDrive 保留下来的接口
- 当前版本已经转向 transformer-style trajectory decoder

### 1.2 `LossComputer` / `loss_bce` 初始化了但几乎没用

位置：

- 第 769-770 行

现状：

- `self.loss_computer = LossComputer(config)`
- `self.loss_bce = nn.BCEWithLogitsLoss()`

但 RL 主链路里没有真正调用它们算最终 loss。

说明：

- 这部分更像 IL 版本遗留

### 1.3 `transfuser_loss` 被 import 但 RL agent 不用

文件：

- [diffusiondrivev2_rl_agent.py](../navsim/agents/diffusiondrivev2/diffusiondrivev2_rl_agent.py)

现状：

- import 了 `transfuser_loss`
- 但 `compute_loss` 实际只读 `predictions["loss"]`

## 2. 参数和调用上的耦合点

### 2.1 `_keyval_embedding` 写死了 `8**2 + 1`

位置：

- [diffusiondrivev2_model_rl.py](../navsim/agents/diffusiondrivev2/diffusiondrivev2_model_rl.py) 第 165 行附近

风险：

- 它默认 backbone 最终 token 网格是 `8x8`
- 如果你换 backbone 或改输入分辨率，可能直接 shape 不匹配

### 2.2 `_bev_downscale` 写死输入通道 `512`

位置：

- 第 169 行附近

风险：

- 强绑定到当前 ResNet34 风格 backbone
- 换成别的 backbone 时要同步改通道

### 2.3 `bev_proj` 写死输入维度 `320`

原因是：

- `256` 来自 token 还原后的 dense feature
- `64` 来自 `bev_feature_upscale`

所以只要任一上游维度改了，这里也要改。

## 3. 代码里和“你直觉以为的 GT”不完全一样的地方

### 3.1 `metric_cache.trajectory` 不是人工监督轨迹

来源：

- [metric_cache_processor.py](../navsim/planning/metric_caching/metric_cache_processor.py) 第 217-229 行

这里缓存的是：

- `pdm_closed_trajectory`

也就是 PDM-Closed planner 产生的参考轨迹。

所以：

- scorer 内部 reference proposal 不是简单的 `targets["trajectory"]`
- 代码注释里有时写“GT”，但更准确说法应该是“PDM reference trajectory”

### 3.2 奖励阈值里的 `reward_gt` 来自 `targets["trajectory"]`

训练时作者又把：

- `targets["trajectory"]`

拼到候选集合最后一列，并把它的得分命名成 `reward_gt`。

结果就是：

- 评分归一化参考来自 `metric_cache.trajectory`
- 正样本筛选阈值又参考 `targets["trajectory"]`

这是一种混合策略，不是单一 reference。

## 4. 可能的可疑实现或值得复核的点

### 4.1 初始 noising timestep 和 rollout timesteps 看起来不完全对齐

代码里：

- 初始加噪 `trunc_timesteps = 8`
- 训练 rollout `roll_timesteps` 大致是 `[18, 16, ..., 0]`
- 测试 rollout 大致是 `[10, 0]`

从静态阅读看，初始 sample 的噪声级别和后续 denoise timesteps 在数值上没有完全自然对齐。

这不一定是 bug，但至少是个值得实验验证的点：

- 作者可能实现的是“截断链的自定义索引”
- 也可能是实验改参后留下的耦合残留

### 4.2 `status_encoding` 参数被一路传下去，但 decoder 层里基本没用

`CustomTransformerDecoderLayer.forward` 接受了 `status_encoding`，但当前主体逻辑没有直接消费它。

这说明：

- 接口是为后续条件扩展预留的
- 当前主实现主要依赖 `ego_query` 和 `time_embed`

### 4.3 `camera_feature`、`token`、`global_img` 在轨迹头里存在弱使用或未使用

现状：

- `camera_feature` 参数一路传到 `TrajectoryHead`，但关键路径里没真正参与计算
- `global_img` 总是传 `None`
- `token` 在 RL 版里基本没参与策略逻辑

这说明接口设计比当前实现更泛化，但也更容易让初学者误判“哪些东西真在起作用”。

### 4.4 `IL_loss_b` 的 batch 维实现值得再确认

在 `get_rlloss` 里：

- `traj_l1.mean()` 是全局标量
- 然后被累加到 `IL_loss_b` 这个按 batch 组织的向量上

结果是：

- 每个 batch 样本最后加到的是同一个全局均值

从工程直觉看，作者本意可能想做 per-batch IL loss，但现在实现更像“批内共享一个平均监督项”。

这点很值得你后续实验时复核。

## 5. 和论文描述对照时要注意的地方

### 5.1 论文讲的是 `scale-adaptive multiplicative noise`

代码里确实实现了乘性噪声，但“adaptive”的部分主要体现在：

- 噪声直接作用在轨迹样本尺度上
- `x / 50`, `y / 20` 的归一化本身也提供了各向异性尺度

它不是特别复杂的显式位置依赖噪声公式，而是偏工程实现版本。

### 5.2 论文讲 `intra-anchor GRPO`

代码中最关键的落地点是：

- 把 reward reshape 成 `[B, num_groups, 20]`
- 在 `dim=1` 上做 mean/std

这正是“同 anchor 内部比较”。

### 5.3 论文讲 `inter-anchor truncated GRPO`

代码没有直接做跨 anchor 的完整 pairwise 排序，而是通过：

- `reward > reward_gt`
- safety 子指标掩码
- negative advantage 截断

实现一个更简单、更工程化的版本。

## 6. 最值得迁移到你自己工程里的 5 个设计思想

### 6.1 用 anchor 把动作空间先分块，再做局部扩散

这比从纯噪声直接搜整个轨迹空间稳很多。

### 6.2 把轨迹条件注入做成“地图局部采样 + agent 交互 + ego 全局条件 + timestep modulation”

这比单纯把所有条件粗暴 concat 更有任务结构。

### 6.3 用闭环 evaluator 做 reward，而不是只用开环 L1

这是从“像数据”转向“像真实驾驶”的关键一步。

### 6.4 保留 IL 辅助项作为 RL 的稳定器

对新手自己做生成式规划尤其重要。

### 6.5 用 Bézier 导数恢复 yaw

这是一个非常实用的小技巧，简单但有效。

## 7. 如果你准备继续深挖，我建议的顺序

1. 先手推一遍 `TrajectoryHead.forward_train_rl`
2. 再对照 `DDIMScheduler_with_logprob.step`
3. 然后读 `get_rlloss`
4. 最后回头看 `pdm_score_para -> PDMSimulator -> PDMScorer`

这样你会先抓住“学习信号从哪里来”，再去看底层 reward 细节，不容易在大量评测代码里迷路。
