# 输出适配、验证与部署路线

## 主定义

输出适配是把 DiffusionDriveV2 `TrajectoryHead` 的原始轨迹类输出固定暴露出来，并在 shadow / switch / fusion 阶段中决定如何使用；第一阶段不重写原有规则链路，也不让模型直接生成 `LocalRoad + Decision` 语义字段。

## 最终记忆版本

`DiffusionDriveV2 只给多模态轨迹和分数；第一阶段 shadow 只记录评估，不改原有规则输出。`

## DiffusionDriveV2 允许输出

```text
selected_trajectory
candidate_trajectories
candidate_scores
valid_flag
```

禁止让 DiffusionDriveV2 直接输出：

```text
centerline_points
left_boundary_points
right_boundary_points
target_stop_s
target_speed
target_parking_slot
nudge_left_boundary
nudge_right_boundary
```

这些字段不属于 DiffusionDriveV2 规划头语义。

## 使用模式

第一阶段采用 shadow：

```text
原 SSM 输入 -> 原有规则黑盒 -> baseline 输出
原 SSM 输入 -> SSMConditionEncoder -> TrajectoryHead -> 轨迹输出
```

shadow 阶段只做：

```text
记录 candidate_trajectories
记录 candidate_scores
记录 selected_trajectory
与 baseline / 回放标签 / 离线指标对齐
```

shadow 阶段不做：

```text
不重写 GenerateLocalRoad 规则
不重写 SDDecider 规则
不逐字段修改 centerline / boundary / stop fence / nudge boundary
不让模型直接填 LocalRoad / Decision
```

第二阶段进入 switch：

```text
TrajectoryHead 输出通过确定性校验 -> 允许使用模型轨迹相关输出
校验失败 -> baseline / fallback
```

fusion 是后续阶段，不作为第一阶段目标。

## 确定性校验

模型输出不能直接影响规划。

必须校验：

| 校验项 | 失败处理 |
|---|---|
| `valid_flag != true` | fallback |
| 输出无 NaN/Inf | fallback |
| `selected_trajectory` shape 合法 | fallback |
| `candidate_trajectories` shape 合法 | fallback |
| `candidate_scores` shape 合法 | fallback |
| 轨迹坐标系与 SSM 局部坐标一致 | fallback |
| 轨迹点在部署 BEV 范围内 | fallback |
| 多帧输出抖动可控 | fallback |
| RoadPlus 指针完整 | 不进入规划 |

## fallback 策略

shadow 阶段：

```text
模型成功 -> 只记录模型输出
模型失败 -> 记录 fallback_reason，不影响 baseline
```

switch 阶段：

```text
E2E 轨迹输出成功且校验通过 -> 允许使用模型轨迹相关输出
E2E 失败或校验失败 -> baseline / fallback
```

更后续阶段：

```text
E2E 成功 -> 使用稳定后的 E2E 输出策略
E2E 失败 -> safe stop fallback
```

## 回放验证路线

### 阶段 A：离线模型验证

输入：固定导出的 `ssm_frame_input.npz`。

检查：

1. condition tensor shape 固定。
2. PyTorch wrapper 能暴露 `candidate_trajectories / candidate_scores / selected_trajectory`。
3. 同一输入多次运行结果一致或随机性被显式记录。
4. selected trajectory 不越界、不异常跳变。

### 阶段 B：Shadow 对齐验证

检查：

1. 模型输出与 baseline 输出时间戳对齐。
2. 模型轨迹坐标系与 SSM 局部坐标一致。
3. selected trajectory 不越过明显 freespace 非法区域。
4. candidate 分数排序稳定。
5. 多帧 selected trajectory 抖动可量化。

### 阶段 C：链路验证

检查：

1. 原 `OUTPUT_SLOT_SSM_LOCAL_ROAD` 持续输出。
2. `GridMapModule` 持续收到 `SsmMsgV2`。
3. `OUTPUT_SLOT_SSM_GRID` 持续输出。
4. `FW_SSM_SYNCHRONIZATION` 能合成 `SsmMsgV2Plus`。
5. `SDPlanningModule` 能进入 `SDPlanner::Run`。
6. 新增 shadow 记录不破坏原有链路。

### 阶段 D：行为验证

检查：

1. 轨迹不跳变。
2. 低速泊车速度合理。
3. 停车点稳定。
4. 障碍物近距离场景下候选轨迹分数有可解释差异。
5. freespace 缺失或异常时能给出 `valid_flag=false`。

## ONNX/HBM 部署路线

推荐顺序：

```text
PyTorch deployment core
  -> torchscript/onnx dry run
  -> ONNXRuntime 对齐
  -> 算子扫描
  -> PTQ 样本导出
  -> hb_compile
  -> HBM runtime wrapper
  -> 端侧回放
```

第一版 HBM 输入建议固定：

```text
ego_query       [1,1,256]
agents_query    [1,30,256]
bev_feature     [1,256,H,W]
status_encoding [1,1,256]
status_feature  [1,8]
valid_masks     [1,30]
```

第一版 HBM 输出建议固定：

```text
selected_trajectory   [1,8,3]
candidate_trajectories [1,G,8,3]
candidate_scores      [1,G]
valid_flag            [1]
```

`LocalRoad + Decision` 的填充不放进 HBM。
