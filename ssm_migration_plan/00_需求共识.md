# 已确认需求共识

## 主定义

本项目的核心需求，是在 J6 现有 SSM module 输入链路不变的前提下，只移植 DiffusionDriveV2 的规划头，用新增编码器把现有 SSM 输入编码为 `TrajectoryHead` 需要的条件张量，并把规划头的原始多模态轨迹输出作为后续工程 adapter 的输入。

## 最终记忆版本

`SSM 原始输入不变 -> SSMConditionEncoder -> DiffusionDriveV2 TrajectoryHead -> 多模态轨迹输出；不接 DiffusionDriveV2 感知 backbone，不要求相机/点云原始输入，不新增 LocalRoad/Decision 预测头。`

## 已明确的边界

### 1. 只移植规划头，不移植全网络

当前目标不是部署完整 DiffusionDriveV2。

明确抛弃：

```text
camera_feature / lidar_feature
  -> TransfuserBackbone
  -> V2TransfuserModel full forward
```

明确保留：

```text
TrajectoryHead
  plan_anchor
  diffusion scheduler / timestep encoding
  plan_anchor_encoder
  diff_decoder
  scorer_decoder
  fine_scorer_decoder
  selector 输出
```

### 2. J6 SSM module 输入链路不变

移植位置是 J6 的 SSM module。

不改变：

```text
topic
input slot
现有 dataflow
SSM 原始输入消息集合
```

依据文档：

```text
/home/yihang/Documents/j5code/端到端SSM替换文档/02_现有输入契约.md
```

第一阶段需要围绕该文档中的现有输入构造模型条件，不引入三前向相机图像、点云或 NAVSIM 原始 BEV raster。

### 3. 需要新增并训练编码器

新增模块的核心是：

```text
SSMConditionEncoder
```

它的职责不是生成 LocalRoad/Decision 字段，而是把 SSM 原始输入编码为 DiffusionDriveV2 规划头输入：

```text
ego_query
agents_query
bev_feature
bev_spatial_shape
status_encoding
status_feature
```

这些张量必须通过训练获得合理分布，不能靠手工常量或规则拼接代替。

### 4. 不强行沿用 SparseDrive adapter 边界

旧链路中的：

```text
det_adapter
map_adapter
ego_adapter
det_instance_feature
det_prediction
map_instance_feature
map_prediction
ego_feature
temp_instance_feature
```

是 SparseDrive motion head 的输入契约，不是 DiffusionDriveV2 规划头的天然输入契约。

本项目允许完全跳出旧 `det_adapter / map_adapter / ego_adapter` 框架。

### 5. 不改变 DiffusionDriveV2 输出语义

明确不新增：

```text
LocalRoad head
Decision head
centerline head
boundary head
stop fence head
nudge boundary head
```

DiffusionDriveV2 规划头只输出轨迹类结果：

```text
candidate_trajectories
candidate_scores
selected_trajectory
valid_flag
```

### 6. 允许 deployment wrapper

允许写 wrapper，但它只做工程化暴露和裁剪：

```text
关闭端侧 PDMScorer / reward_dict 路径
固定 deployment forward 输入输出
暴露 forward_test_rl 内部已经生成的 traj_to_score / selected trajectory
```

不允许 wrapper 改变规划头语义或新增预测头。

### 7. 原有规则链路先作为黑盒

原有规则部分不作为当前讨论重点。

第一阶段不重写、不优化、不逐字段追问原有规则逻辑。合理工程形态是：

```text
原 SSM 输入 -> 原有规则黑盒 -> baseline 输出
原 SSM 输入 -> SSMConditionEncoder -> DiffusionDriveV2 TrajectoryHead -> 轨迹输出
```

后续在 shadow / switch / fusion 模式中决定是否使用模型轨迹输出。

### 8. 第一阶段推荐模式

第一阶段先做 shadow：

```text
模型轨迹只记录、对齐、评估
不影响原有输出
```

稳定后再进入 switch：

```text
在确定性校验通过时使用模型轨迹相关输出
失败时回退 baseline
```

fusion 作为更后续阶段，不作为第一阶段目标。
